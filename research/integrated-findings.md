## 0. 한 줄 결론
세 트랙이 **하나의 킥으로 수렴**한다: **"로그인 없이 30초, 입력 3개로 — 내가 놓치고 있던 숨은 혜택을 '왜 해당되는지' 근거와 함께 발견하고, 신청까지 한 화면에서."** intake의 A(정확한 자격 매칭)·B(신청 안내)·C(숨은 혜택 발견) 세 각도는 경쟁하지 않고 **한 흐름으로 합쳐진다.**

## 1. 문제 (JTBD에서) — "왜 못 받나"
- **1차 통증 = "몰라서 못 받는다."** 한국 복지는 *신청주의* 구조라 제도가 많아도(복지멤버십 안내 대상만 128종, 중앙+지자체+민간 분산) 자기 가구 맞춤 *전체 목록*을 못 본다. `[jtbd-2]`
- **2차 통증 = 자격 오판 자기검열.** "기준이 엄격해 신청해도 안 될 것 같다"며 시도조차 안 함. `[jtbd-3, unverified 수치 포함]`
- **3차 니즈 = 행동 전환.** 목록만으로 부족 — "어디서·어떻게 신청"까지 이어져야 함.
- 페르소나 4개(신혼·영유아 / 청년 1인 / 노년·기초연금 / 위기가구) 모두 기존 대안(맘카페·토스·주민센터·자녀 대리검색)이 **지역·시점 불일치**로 실패. `[jtbd-1,4]`

## 2. 이미 있는 것 (국내) — "무엇을 새로 만들지 말까"
- "입력 → 혜택 목록 → 신청" 흐름 자체는 **이미 존재**: 복지로(모의계산·맞춤형급여안내), 정부24 보조금24, 민간(토스, 2024.8 종료). `[dom-1,2,3,4]`
- **공통 약점(= 우리의 빈틈):**
  1. **로그인·본인인증·정보연계 동의가 전제** → 즉시 체험 불가 `[dom-2,3]`
  2. 비로그인 모의계산은 **사업별로 쪼개짐** `[dom-1]`
  3. 결과가 **"왜 해당되는지" 설명 없이** 목록 나열
  4. 중앙+지자체 혜택이 **채널별로 흩어짐** `[dom-6]`
- 규제: 자격 *확정*은 행정청 권한 → **"참고용 자가진단(확정 아님)" 명시 필수**. 비로그인·세션 휘발 입력으로 개인정보 리스크 회피. `[dom-5]`

## 3. 검증된 메커니즘 (해외 ≥3 → 6개 확인) — "무엇을 이식할까"
ACCESS NYC, GetCalFresh, mRelief, Turn2us(영국), SupportGoWhere/LifeSG(싱가포르), USA.gov Benefit Finder — 6개 모두 웹 검증. `[ovs-1~6]`

**이식 가능한 킥 (우선순위):**
1. **자격요건의 rules-as-code화 + 데이터/UI 분리** (ACCESS NYC, mRelief) — 혜택의 선정기준을 JSON 규칙으로 만들고 단순 술어 함수로 매칭. **혜택 추가 = 데이터 추가**라 4시간 데모에 최적이며 "정확도 Wow"의 근거. `[ovs-1,3]`
2. **질문 최소화(3~5개, 전부 드롭다운/라디오)** (mRelief, SupportGoWhere) `[ovs-3,5]`
3. **"찾기 → 신청" 한 동선** — 카드에 신청채널·필요서류·한 줄 신청법 (GetCalFresh, Turn2us) `[ovs-2,4]`
4. **익명·무저장 클라이언트 계산 + "추정이지 보장 아님" 면책** (Benefits.gov) — 서버/DB 불필요(4시간 적합) + 공공 신뢰 + "지어내지 않기" 제약 동시 충족. `[ovs-6]`
5. **"왜 묻나요/왜 해당되나요" 도움말**로 근거 가시화 (Turn2us) `[ovs-4]`

## 4. 구현 현실성 (가장 중요한 제약!)
- **4시간 데모 가능.** 핵심 경로: 가구·지역·소득 입력 폼 → **클라이언트 측 규칙 매칭** → 대표 혜택 10~15개 카드 + 신청 링크. `page.tsx`의 select 데모 + `regions.ts`를 입력 폼으로 승격. `[feas-1]`
- ⚠️ **치명적 발견:** 권위 데이터셋(복지로 15083323)을 받을 수는 있으나(로그인 불필요, 367행) **소득·가구·연령 등 자격요건 필드가 없다**(서비스명·요약·소관부처·링크뿐). `[feas-1]`
  - → **자격 매칭은 손으로 큐레이션한 대표 혜택 12~15개 fixture(명시적 eligibility 규칙 포함)에 의존**하고, CSV는 *정식 명칭·복지로 링크 보강용*으로만 사용. 화면에 **"예시 데이터" 명시**.
- 보조금24 API(15113968)는 키 신청·실시간 → **데모에서 전면 배제.** `[feas-2]`
- **재사용 자산:** `web/` 스캐폴드, `regions.ts`(지역 입력), KRDS(UI), `build-shelters.mjs`(CSV 가공 패턴). `shelters.ts`·`KakaoMap.tsx`는 **미사용**(비위치 기반) → 떼어냄.

## 5. 심사위원 방어 (차별화)
- **가장 강한 의심: "복지로/보조금24에 이미 있잖아요?"** `[judge-1,2]`
- **방어 = 무기:** 기존 서비스는 로그인·정보연계 동의가 전제. 우리는 **무로그인·무저장·30초·입력 3개**로 그 마찰을 제거 → "차별화 + 개인정보 의심"을 한 번에 방어.
- **데모 킥:**
  - (A) 입력 직후 **"놓치고 있던 혜택 N건"이 배지로 튀어나오는 순간** (감정적 임팩트)
  - (B) **로그인 화면이 한 번도 안 나오는 것**을 타이머로 증명 (마찰 제거 증명)
- **기억에 남는 한 문장 후보:** *"검색하면 나오는 게 아니라, 안 찾아도 놓친 걸 알려줍니다 — 로그인 없이 30초에."*

## 6. 킥(차별점) 후보 — Stage 02 입력
> 여기서 **확정하지 않는다.** Stage 02에서 발산·수렴·사용자 승인.

- **킥-1 (메인 후보, 강한 수렴):** *무로그인 30초 · 입력 3개 → "놓친 숨은 혜택" 발견 + "왜 해당?" 근거 배지 + 신청 한 화면.*
  - 메커니즘: rules-as-code(JSON 규칙) × 큐레이션 fixture 12~15개 × 클라이언트 매칭 × 익명·무저장.
  - 근거: 국내 빈틈(로그인 마찰·설명 부재) + 해외 검증 메커니즘 + 심사위원 의심 직접 방어.
- **킥-2 (보조 각도):** "왜 해당되는지" 설명을 전면에 — *근거 가시화*가 신뢰의 핵심 (Turn2us형).
- **킥-3 (보조 각도):** 중앙+지자체 **통합** 한 화면 (국내 분산 약점 공략) — 단 데이터 범위 주의(fixture 한계).

## 7. 다음 단계(02)가 알아야 할 위험
1. **데이터 한계가 컨셉을 규정한다:** 자동 전수 매칭은 불가 → "큐레이션 대표 혜택 + 예시 데이터 명시"가 정직한 범위. 02에서 "전국 모든 혜택"으로 과장 금지.
2. **자격 판정 책임 경계:** "참고용 추정, 확정은 기관" 면책 필수(법적·신뢰).
3. **차별화는 '무로그인 + 근거 가시화'에 있다** — 단순 "조건 입력→목록"만 남기면 레드오션.

## 8. 교차검토(LLM Review) 결과 — 종합 "조건부 통과"
독립 리뷰어 검토. 리서치 자체는 출처 검증·데이터 정직성·구현 현실성 측면에서 견고(관점 1·3·5 PASS). 단 **Stage 02에서 반드시 해소할 조건 2건**:

### 🔴 (필수) AI Leverage = 현재 FAIL — 컨셉에 AI가 0건
- **문제:** rules-as-code(JSON 규칙 + 술어 매칭)는 **AI가 아니다.** 현재 설계는 AI 없이 100% 동작 → 공공 AI 해커톤 심사에서 치명적.
- **해소 방향(정당하고 제약 안전한 LLM 2곳):**
  - **(a) 자연어 입력 정규화:** "39살 혼자 살고 월 200 벌어요" → 구조화 입력(가구유형·소득구간) 변환. 판정은 여전히 결정론적 규칙엔진 → "지어내지 않기" 충돌 없음.
  - **(b) 근거 설명문 생성:** 매칭된 혜택마다 "한부모·중위소득 52%여서 해당"을 fixture 규칙값으로부터 자연어로 풀어줌. **단, LLM은 fixture에 있는 사실만 서술하도록 제약**(템플릿+값 주입). 킥-2(근거 가시화)와 직결 → 데모 Wow 강화.
  - ⛔ **자격 *판정* 자체는 절대 LLM에 맡기지 말 것**(환각=거짓 자격 안내=신뢰 붕괴).

### 🔴 (필수) "놓친 N건" 프레이밍 정직화
- **문제:** fixture 12~15개만 가진 상태에서 "당신이 N건 놓쳤다"는 **전수 비교 암시 → 거짓 위험.** 전체 모수를 모르는데 "놓친 건수" 단정 불가.
- **해소:** "**해당될 수 있는 혜택 N건 발견**"(발견 프레이밍) 또는 "예시 N건 중 M건 해당"으로 모수 노출 + 화면에 "예시 데이터(전체 아님)" 배너 상시.

### 🟡 (권장) 범위 수렴
- 킥을 **킥-1 하나로 수렴**(킥-2 근거배지·킥-3 통합은 킥-1의 요소로 흡수). fixture는 출처·eligibility 규칙·신청링크 갖춘 **8~10개로 시작** 후 시간 남으면 확장.
- 해외 사례의 *메커니즘*은 이식하되 *규모*(40~1,000개 프로그램)는 베끼지 말 것 — 표면 베끼기 위험은 "건수"에 있다.
- GetCalFresh 등 해외 *실적 수치*를 우리 효과인 양 전용 금지.
