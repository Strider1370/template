## 1. Theme Analysis (주제 해석)

**표면적 의미**: 임산부·신생아 대상 정부 지원을 한곳에서 안내하는 서비스.

**주최 측이 실제로 기대하는 가치**: 저출생 대응 맥락에서, *존재하지만 국민이 못 받고 있는* 지원의 **수혜율(take-up)**을 끌어올리는 것. 즉 "제도를 더 만들기"가 아니라 "있는 제도를 제때·빠짐없이 받게 하기".

**대부분 팀이 선택할 접근(흔한 것 5+, 피해야 함)**:
1. 임신 주차별 정보 제공 앱 → **이미 존재: 아이마중**(복지부, 주차 3~40·월령 1~36) `domestic-src-05`
2. 정부지원 안내 챗봇 → **이미 존재: 129·복지로 상담** `domestic-src-10`
3. 자격 모의계산기 → **이미 존재: 복지로 모의계산** `domestic-src-07`
4. 지역별 출산지원금 비교표 → 블로그·기사 다수 `judge-src-07`
5. 통합 신청 폼 → **이미 존재: 정부24 맘편한임신/행복출산** `domestic-src-01` `jtbd-src-02`

**기술적으로 화려하지만 가치가 약한 방향**: 풀 AI 상담 챗봇(환각 리스크 + 차별성 약함 + 129와 중복). → AI는 *절제*해서 써야 가점(트랙 E).

**공공성·실현가능성·시연가능성 균형**: 공공성 高(수혜율), 4시간 시연 可(고정 데이터+계산 로직), 시연성 高(입력→타임라인 즉시 변화).

## 2. 핵심 문제(JTBD) 종합

세 가지 마찰이 순차적으로 누적된다 (`claim: problem-01/02/03`):

- **P1 — 정보 산재**: 정부24(행안부)·복지로(복지부)·아이사랑·129·지자체로 입구가 갈라져, 사용자가 "내가 뭘 받을 수 있는지" 전체 그림을 못 본다. `jtbd-src-05` `jtbd-src-06`
- **P2 — 시기 마감 절벽 (가장 아픈 곳)**: 신청 창이 좁거나 소급 마감이 있어 **놓치면 곧 돈**. 부모급여·첫만남은 출생 60일 이내 소급 `jtbd-src-03`, 산모·신생아 건강관리는 분만예정일 40일 전~출산 30일 후 `feasibility-src-06`.
- **P3 — 자격·지역 판단 어려움**: 소득·태아수·거주지·질환에 따라 받을 제도가 달라지고, 특히 출산지원금은 **지자체별 편차 극심** `domestic-src-06` `judge-src-07`.

**대표 JTBD**: "임신/출산이라는 큰 이벤트 앞에서, 내 상황(주차·지역·소득)에 맞는 지원을 *제때, 하나도 안 놓치고* 받고 싶다." 현재 대안(맘카페·검색·정부24)은 push가 아니라 pull이고 시간축이 없어 충분치 않다.

## 4. 해외 검증 메커니즘 (이식 가능, ≥3)

8개 사례가 3개 메커니즘으로 수렴 (`claim: overseas-01`):

1. **라이프이벤트 타임라인 + 기한 노출** — 영국 Sure Start: "출산 11주 전~후 6개월" 클레임 윈도우를 1급 정보로 제시 `overseas-src-10`. 호주: 출산 전 3개월 사전신청 + 시기별 할 일 `overseas-src-11`.
2. **최소 질문 자격 사전 스크리닝** — mRelief(최소질문 SNAP 사전판정) `overseas-src-07`, USAGov Benefit Finder(익명 질문→1,000+ 매칭) `overseas-src-08`, GOV.UK 자격 체커 `overseas-src-09`.
3. **Proactive 능동 안내** — 에스토니아 신청불필요 자동 통지 `overseas-src-01` `overseas-src-02`, 캐나다 ABA(출생등록에 신청 끼워넣기) `overseas-src-05`, 싱가포르 LifeSG Baby Bonus `overseas-src-03`.

## 5. 구현 현실성 (4시간)

- **데이터**: 정부 지원 제도 = **고정 JSON 8~12개**(임신·출산진료비 바우처, 첫만남이용권, 부모급여, 아동수당, 산모·신생아 건강관리, 고위험임산부 의료비 등). 키트에 기존 자산 없음 → **신규 제작이 크리티컬 패스**. 금액 등 일부 수치는 2차출처 → `verified` 플래그로 투명 처리, 확정 전 재검증.
- **실 API(혼합 전략)**: 카카오 로컬 키워드 검색(가까운 보건소·육아종합지원센터·산후조리원). **주의: JS 지도키와 별개의 REST 키(`KakaoAK` 헤더) + Next.js 서버 프록시 필요** `feasibility-src-09`. 폴백 = 고정 시설 목록.
- **AI 레이어**: 서버사이드 Claude 1콜에 제도 JSON 전체를 컨텍스트로 주입(환각 방지) → 자격 자연어 Q&A + 개인 상황 맞춤 요약. 키 없으면 규칙 기반 요약 폴백.
- **재사용 자산**: `regions.ts`(거주지 셀렉터), `shelters.ts` Haversine(시설 정렬), `KakaoMap.tsx`(키없을때 fallback), KRDS.
- **최소 데모 happy path**: 입력(주차/예정일/지역) → 시간축 정렬 → "지금/곧" 제도 카드 + 규칙기반 브리핑. **카카오·Claude 키가 모두 없어도 동작**.
- **가장 위험한 외부 의존성**: (1) 제도 데이터 정확성/최신성, (2) 카카오 REST 키 도메인 설정.

## 6. 심사위원 관점 (압박 테스트)

- **첫 30초에 이해시킬 것**: "흩어진 임신·출산 지원을, 내 주차·내 동네 기준으로 *지금 뭘 신청해야 하는지* 시간순으로 정렬해 준다."
- **가장 강한 의심 → 방어**:
  - "정부24가 이미 하지 않나?" → 정부24=신청 **창구**, 우리=언제·어디로 갈지 판단하는 **내비게이션 레이어**. `domestic-src-01`
  - "제도 데이터 최신성은?" → 출처·확인일자 명시 + `verified` 플래그 투명 공개.
  - "AI가 꼭 필요한가, 규칙 엔진 아닌가?" → 정렬은 규칙, AI는 *비정형 상황 매핑·평이체 재서술*에만 절제 사용(가점 포인트).
  - "오안내 책임은?" → 자격 *판단 보조*까지, 최종은 공식 창구 링크로 연결.
- **데모 킥**: 입력값(주차/지역) 바꾸면 **타임라인이 실시간 재정렬 + 지자체 지원금이 즉시 바뀜**.
- **기억에 남는 한 문장(후보)**: "있는 지원을 못 받는 게 아니라, *언제 받는지*를 몰라서 놓친다."

## 7. 차별점(킥) 후보 — Stage 02 선택용 (확정 아님)

- **킥 후보 ①  D-day 타임라인 + 마감 손실 경고** *(가장 유력)*: 주차/예정일 기준 제도를 시간축에 배치, "지금 신청 가능(마감 D-N)" → "곧 열림" 정렬. 60일 소급·40일전~30일후 창처럼 *놓치면 돈 잃는* 제도를 카운트다운으로 경고. → P2를 정면으로 푼다. 근거: `jtbd-src-03` `feasibility-src-06`, UK/호주 메커니즘.
- **킥 후보 ②  지역 스위치(지자체 자동 매칭)**: 거주지 바꾸면 지자체 출산지원금이 실시간 변동. 데모에서 즉각적 'Wow'. 근거: `judge-src-07` `domestic-src-06`.
- **킥 후보 ③  절제형 AI 자격 내비게이터**: 비정형 상황(쌍둥이/고위험/전입 3개월)을 자연어로 물으면 고정 제도 JSON 위에서 해당 제도만 추려 평이체 요약. mRelief 스크리너의 LLM 업그레이드. 근거: `overseas-src-07` `overseas-src-08`.

> **권장(메인 의견, 비확정 — 교차검토 반영)**: Wow는 **1개로 압축**한다. **① D-day 타임라인 + 마감 손실 경고**를 단일 Wow로, **②지역 스위치는 같은 화면 안의 보조 동작**, **③AI 자격 내비게이터는 "있으면 가산점"(키 없으면 규칙 폴백)**으로 명시적 강등. → 4시간 데이터 큐레이션(크리티컬 패스)과 충돌 방지. 단 **최종 인사이트·Wow·범위는 Stage 02에서 사용자 승인**.
