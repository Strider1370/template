# 트랙 D — 구현 현실성 (feasibility)

## 1. 한 줄 결론

4시간 데모 **가능**. 가장 짧은 핵심 경로 = **3~5문항 입력 폼(가구 구성·지역·소득) → 클라이언트 측 규칙 매칭 → 대표 혜택 10~15개를 카드로 출력 + "신청 방법/바로가기" 링크**. 전체 복지제도 자동 매칭은 버리고 **손으로 큐레이션한 자격 규칙 fixture**로 가는 것이 핵심 결정.

---

## 2. 데이터 확보 시나리오 (성공 / 폴백)

이 환경은 `data.go.kr`이 **열려 있음**(WebFetch로 15083323 / 15113968 페이지 정상 응답 확인). 단, 데이터의 *구조*가 더 큰 문제다.

| 시나리오 | 무엇을 받나 | 가능성 | 한계 / 비고 |
|---|---|---|---|
| **성공 A (권장 폴백 아님, 실제 1순위)** | 복지로 복지서비스정보 CSV `15083323` (367행) | **높음** — 로그인 불필요로 확인됨 ("파일데이터는 로그인 없이 다운로드") | **자격요건/소득/가구 필드 없음.** 컬럼 = 서비스ID·명·URL·요약·소관부처·연도뿐. → 그대로는 **매칭 불가**, 혜택 "목록·설명·링크"용으로만 사용 |
| 성공 B | 보조금24/중앙부처복지 API `15113968`·`15090532` | 낮음(당일) | API 키 신청 필요(자동승인이라도 지연 위험). 상세 자격은 **서비스별 상세조회 API 1건씩** 호출 구조 → 4시간 내 자격 스키마 정규화 비현실적 |
| **폴백 C (실질 채택안)** | **손으로 큐레이션한 대표 혜택 12~15개 JSON fixture** (각 항목에 명시적 eligibility 규칙: 가구유형/소득분위/연령/지역) | **확실** | 출처를 복지로 실제 제도(예: 청년월세, 기초연금, 아동수당, 에너지바우처 등)에 두고 화면에 "예시 데이터" 명시. 4시간 매칭 엔진은 여기에만 의존 |
| 폴백 D | 성공 A의 367행 CSV를 **fixture에 메타(정식 명칭·소관부처·복지로 URL) 머지** | 중 | C의 신뢰도·"신청 바로가기" 정확도를 올림. 시간 남으면 |

**결론: 367행 CSV는 받되(라벨·링크·신뢰도용), 자격 매칭은 큐레이션 fixture(C)로 한다.** CSV 단독으로는 "받을 수 있는지" 판정이 불가능하기 때문 — 이것이 이 주제 최대 함정.

## 3. 필요한 기술·구현 요소 + 4시간 현실성

### Must (반드시)
- **입력 폼**: 가구 구성(1인/부부/한부모/다자녀/노인 등 선택), 거주 시도·시군구(`regions.ts` 그대로), 소득(월 가구소득 또는 "중위소득 %" 슬라이더). KRDS native `<select>`/input.
- **규칙 매칭 엔진**: fixture 각 혜택의 `eligibility: { household?, ageMin/Max?, incomeMaxPct?, regions? }` ↔ 입력값 비교하는 **순수 함수 `match(input, benefit) → boolean`**. 클라이언트 단, API 불필요.
- **결과 카드 리스트**: 매칭된 혜택 → KRDS 카드(혜택명·요약·지원금액·`Badge`로 "신청가능"·신청처 링크).
- **혜택 fixture JSON**: 12~15건, 출처 명시.

### 축소 (할 수 있으면 좋지만 데모 핵심 아님)
- 시군구 단위 지자체 혜택(전국 공통 중앙부처 혜택만으로 충분). 지자체별은 `regions=['서울특별시']` 식 1~2건 샘플만.
- "놓치고 있던 혜택 N건" 강조(notes의 angle C) — 매칭 결과에 카운트 배지만 붙이면 저비용 Wow.
- 신청 동선 상세(angle B) — 카드에 "복지로/정부24 바로가기" 외부 링크면 충분.

### 위험 (4시간 안에 하지 마라)
- 367행/수천건 전체 제도 자동 자격판정. **소득 환산(중위소득 가구원수별 표), 재산 기준, 중복 수급 배제** 등 실제 복지 자격 규칙은 매우 복잡 → 절대 풀지 말 것.
- 상세조회 API 정규화, 실시간 호출.

## 5. 가장 위험한 의존성 Top 3 + 대응

1. **[데이터 구조] 권위 데이터셋에 자격요건 필드가 없다 (최대 위험).** 15083323 CSV는 설명·링크뿐, 소득/가구/연령 기준 없음 → "받을 수 있는 혜택" 판정 불가.
   → **대응:** 자격 매칭은 큐레이션 fixture(폴백 C)에 100% 의존. CSV는 명칭·소관부처·복지로 링크 보강용으로만. 화면에 "예시 데이터" 명시(intake 제약 "지어내지 않기" 충족).

2. **[규칙 복잡성] 자격 판정 규칙의 실제 복잡도.** 중위소득 가구원수 환산·재산기준·중복배제는 4시간 밖.
   → **대응:** fixture 규칙을 **단순 임계값**(가구유형 매칭 + 연령 범위 + 소득 상한 %)으로 의도적 축소. 발표에서 "데모는 핵심 조건만 판정, 정확 판정은 복지로 연계" 라고 솔직히 프레이밍.

3. **[당일 API] 보조금24/상세조회 API 키·스키마 의존.** 키 지연·서비스별 호출 구조로 시간 증발 위험.
   → **대응:** 실시간 API **경로 자체를 데모에서 배제.** 전부 정적 fixture + 클라이언트 매칭. 외부는 "신청 바로가기" 링크만(네트워크 실패해도 데모 안 죽음).
