# feasibility.md — 트랙 D: 데이터·구현 현실성

> Stage 01 병렬 리서치 / 주제: "민원·행정정보 접근성을 높이는 챗봇 또는 검색 서비스"
> 방향 가중치: **B(절차 안내형) · C(쉬운 말 번역형)**. (A 맞춤 추천형은 보조)
> 작성: 트랙 D · 2026-06-20 · 웹 접근 일부 가능(WebSearch/Fetch 동작), data.go.kr 직접 다운로드는 미시도(막힌 환경 가정).

---

## 0. 한 줄 결론 (현실성 판단)

**B+C는 4시간 안에 확실히 시연 가능하다.** 단 핵심은 "데이터를 얼마나 많이 넣느냐"가 아니라 **권위 있는 작은 fixture(20~50개 민원/혜택) + LLM RAG(근거 인용)** 의 조합이다.
가장 위험한 의존성은 **(1) data.go.kr 당일 다운로드(막힐 수 있음)** 와 **(2) OPENAI_API_KEY 유무**. 둘 다 **사전 확보 + 폴백**으로 0에 가깝게 만들 수 있다. C(쉬운 말 번역)는 데이터 의존이 가장 작아 **데모 백업으로 최강**이다.

## 1. 핵심 경로 최단화 (4시간 안에 도는 최소 데모)

두 방향 모두 **단일 페이지 웹앱(채팅/검색 한 화면)** 으로 충분하다. 키트 `web/` 스캐폴드 + `/api/llm` 라우트가 이미 두뇌를 제공하므로 새로 만들 코드가 적다.

### B(절차 안내) 최단 경로
1. 사용자가 자연어로 질문("이사했는데 전입신고 어떻게 해요?") 또는 민원 목록에서 선택.
2. `/api/llm`에 **fixture(민원 절차 JSON)에서 키워드 매칭으로 추린 후보 1~3건**을 컨텍스트로 넘김(RAG).
3. LLM이 **관할기관·구비서류·처리기간·신청방법·다음 단계**를 카드로 정리하고 **출처(어느 민원사무)** 를 인용.
4. (선택) `regions.ts`로 시도/시군구 선택 → "내 지역 관할 기관" 문구 보강.
- 최소 데모: 1)~3)만으로 Wow 성립. 4)는 시간 남으면.

### C(쉬운 말 번역) 최단 경로
1. 사용자가 **어려운 공문/안내문을 붙여넣기**(textarea).
2. `/api/llm`이 ①쉬운 말 요약 ②어려운 용어 풀이(용어사전 fixture로 근거) ③"내가 뭘 해야 하나" 체크리스트로 변환.
3. 결과를 좌(원문)·우(쉬운 말) 대비 카드로.
- 최소 데모: textarea + LLM 1콜. **데이터 의존이 가장 작다(용어사전 없이도 동작, 있으면 신뢰 ↑).**

> **권장 핵심 경로:** C를 "확실히 도는 바닥"으로 깔고, B를 "Wow 화면"으로 얹는다. 두 화면이 같은 `/api/llm`·같은 RAG 패턴을 공유하므로 구현 중복이 거의 없다.

**중요한 현실(실측·검증):**
- **정형 "민원 절차/구비서류" 단일 API는 깔끔하게 없다.** 정부24의 공식 오픈 API는 대부분 **통계연보** 또는 **보조금24 서비스 목록**이다(D2/D6). 민원별 절차 본문(구비서류·처리기간)은 **민원처리기준표(D1)** 에 있으나 관보/HTML 형태라 **정형 적재가 자동화되지 않는다** → 손으로 큐레이션해야 한다.
- 따라서 **B의 데이터는 "전수 API"가 아니라 "권위 출처에서 손으로 정리한 fixture"** 가 현실적이다. 이게 품질 저하가 아니라, 이 도메인의 데이터 성격(산문·법령) 때문이다 — `data/data-sources.md`의 실측 교훈("정밀 자격기준은 대부분 산문이라 자동 전수 판정 불가 → 큐레이션 fixture 병행")과 일치.

## 3. 데이터 입수 현실성 & 전략

**원칙(키트 규칙):** 모집단은 전수 적재(검색·필터·매칭 대상은 전체), 화면 표시만 일부. 시간 부족 시 **데이터가 아니라 화면 수**를 줄인다.

- **막힌 환경(이 클라우드)**: `data.go.kr` 403 가능 → **헛시도 금지.** 사람이 미리 받아 `data/`에 둔 파일 사용. 없으면 사용자에게 **구체 파일 요청**(아래 "사전 확보 요청").
- **B 데이터 전략**:
  - **전수 후보군**으로 D2(보조금24 스냅샷 `data/snapshots/gov24-services.json`, ~10,957건)를 **검색·매칭 모집단**으로 깐다(키워드/소관 필터).
  - **절차 정밀 카드**(구비서류·처리기간·다음단계)는 **민원처리기준표(D1) 기반 큐레이션 fixture 20~40건**(전입신고·출생신고·여권·주민등록등본·자동차 등록 등 최빈 민원). 양식은 `data/welfare/curated-benefits.example.json` 패턴 재사용.
  - → "전수에서 찾고(D2), 정밀하게 답하는(D1 fixture)" 2층 구조. 모집단은 전수, 표시는 일부.
- **C 데이터 전략**: D3(공통표준용어 CSV)를 **용어 풀이 사전 모집단**으로 적재. 없어도 LLM 단독으로 동작하므로 **데모 위험이 가장 낮다.** 용어사전이 붙으면 "근거 있는 풀이"로 신뢰 ↑.
- **금지**: 못 받은 데이터 수치 날조 금지. 못 받았으면 화면·발표에 **"샘플/예시"** 명시.

### 사전 확보 요청 (사용자에게 — 막힌 환경 대비, 우선순위 순)
1. **(C·가장 쉬움)** `data.go.kr/data/15156379` 공통표준용어 CSV 1개 → `data/terms/`.
2. **(B 모집단)** 보조금24 스냅샷이 레포에 없으면 `node data/scripts/dump-gov24.mjs`로 키 있을 때 덤프, 또는 `15113968` JSON 수동 다운로드 → `data/snapshots/`.
3. **(B 정밀)** D1 민원처리기준표에서 **최빈 민원 20~40건**을 발췌(없으면 정부24 각 민원 안내 페이지로 손 정리). 이건 데이터라기보다 큐레이션 작업.

## 4. 실시간 API 필요한가? → 거의 불필요

**이 주제는 실시간 API가 사실상 필요 없다.** 민원 절차·구비서류·용어는 **자주 안 바뀌는 정적 데이터**다(분기·연 단위 개정). → **정적 fixture/스냅샷으로 100% 대체 가능.**

- 실시간성이 필요한 유일한 후보는 "특정 기관 실시간 대기인원/예약" 같은 것인데, **데모 범위 밖**(고위험·저보상). 넣지 마라.
- 외부 의존은 **LLM API 한 곳뿐**. 그래서 폴백 설계가 핵심(§6).

## 7. LLM + RAG 신뢰 설계 메모 (이 주제의 승부처)

행정정보는 **틀리면 안 되는** 도메인이라 "그럴듯한 LLM 답"은 오히려 위험하다. 신뢰는 **근거 인용**으로 푼다:

1. **검색→근거 선별(R)**: 사용자 질문을 fixture(B: 절차/D2 서비스, C: 공통표준용어)에서 키워드/임베딩 없이 **단순 키워드 매칭으로 1~5건** 추림(4시간엔 임베딩 불필요, 키워드로 충분).
2. **근거 주입(A)**: 추린 항목을 JSON 컨텍스트로 user 메시지에 합쳐 `/api/llm` 호출. system: *"아래 근거 안에서만 답하라. 근거에 없으면 '공식 안내 확인이 필요합니다'라고 말하고 지어내지 마라. 답변 끝에 사용한 근거 항목명을 인용하라."*
3. **생성+인용(G)**: 카드 하단에 **"근거: 〈전입신고〉 민원처리기준표 / 〈[용어]〉 공통표준용어"** 처럼 출처 칩 노출. 클릭 시 원문/링크.
4. **JSON 모드 활용**: `json:true`로 구조화 응답(필드: 요약·서류·기관·기간·다음단계·근거ID) → UI 카드에 안정적으로 바인딩, 환각 표면적 축소.
5. **폴백 정직성**: 키 없을 때(`provider:'fallback'`) C는 규칙 치환, B는 fixture 원문 그대로 표시 — **"AI가 없어도 데이터는 진짜"**.

> 핵심 메시지(발표용): "검색은 많지만 **뭐가 맞는지** 모른다(intake 가설 ⑦). 우리는 **권위 출처를 근거로 인용**하는 챗봇으로 그 신뢰 문제를 푼다." — B는 ①④⑤, C는 ②⑥, 둘 다 ⑦을 정조준.
