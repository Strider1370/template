# 🏆 AI 해커톤 협업 워크플로우 (4시간)

> **목표:** 바이브 코딩 중심으로, 두 사람의 강점을 합쳐 4시간 안에 "데모가 확실히 돌아가는" 결과물을 만든다.

---

## 👥 우리 둘의 역할

| | **A (바이브 코딩 담당 / 운전자 🚗)** | **B (코드 읽기 담당 / 정비공 🔧)** |
|---|---|---|
| 강점 | AI 오케스트레이션, 빠른 프로토타이핑 | 코드 독해, 디버그, 기술 현실성 판단 |
| 메인 역할 | 기획·AI 지시·전체 흐름 주도 | 검토·디버그·스펙 검증 |
| 키보드 | **A가 잡는다** (AI 명령 입력) | 옆에서 코드 읽고 포인트 찍어줌 |

**핵심 원칙:** 운전(기획·오케스트레이션)은 A, 정비(검토·디버그)는 B.
둘 다 AI로 작업하되, B는 "AI가 만든 걸 검증하는 고급 역할"을 맡는다.

---

## 🛠️ 사용 도구

- **Claude (Opus)** — 기획, 브레인스토밍, 문서 작성, 검토, 발표자료
- **Codex** — 실제 구현 (서브에이전트 병렬)
- **서브에이전트 병렬 실행** — 리서치 / 구현 속도 확보
- **검증** — Playwright 또는 Claude/Codex 앱 미리보기 브라우저 + 직접 실행

---

## 📋 전체 워크플로우 (단계별)

### 1단계. 기획 & 브레인스토밍
- Claude(Opus)에게 주제를 던지고 함께 아이디어 뼈대를 잡는다.
- **동시에** 서브에이전트를 병렬로 돌려 관련 아이디어/레퍼런스를 검색·수집.
- 👉 **주도: A** / B는 옆에서 "이거 구현 현실성 있나?" 의견.

### 2단계. 스펙 문서 작성 & 검토
- 모인 아이디어를 Claude에게 **스펙 문서**로 정리시킨다.
- 스펙 문서를 **Claude + Codex가 각각 교차 검토** → 결과를 Claude에 전달해 수정.
- 👉 **여기서 B 투입:** B가 사람 눈으로 "4시간 안에 진짜 가능?"을 함께 검토.
  → AI 2개 + 사람 1명 = 3중 검토로 범위를 현실적으로 깎는다.

### 3단계. 구현 계획 문서 작성 & 검토
- 스펙 문서를 바탕으로 Claude에게 **구현 계획 문서**를 만들게 한다.
- 다시 Claude + Codex 교차 검토.
- **동시에 A는 사전 준비 진행** (API 사용 신청, 키 발급, 계정 세팅 등).
- 👉 **주도: A** / B는 계획의 기술적 함정 점검 (예: "이 API 인증 오래 걸림").

### 4단계. 구현 (Codex 병렬)
- Codex에게 서브에이전트 병렬로 구현 시작 명령. **단계별 검증은 최소화**해서 속도 우선.
- **동시에 두 갈래로 작업:**
  - **A:** 발표자료(HTML PPT) 작성 — 스펙 문서 기반 초안.
  - **B:** 구현되는 코드를 실시간으로 읽으며 "스펙과 다르게 가는 부분" 조기 발견. ⭐
  - → 6단계까지 안 기다리고 버그를 미리 잡는 게 시간 절약의 핵심.

### 5단계. 발표자료 1차 작성
- Claude와 함께 HTML 형식 발표자료 초안 작성 (스펙 기반).
- 실제 구현 완료 후 내용을 보강할 예정.
- 👉 **주도: A**

### 6단계. 구현 검증 (스펙 대비)
- Claude/Codex에게 스펙 문서대로 구현됐는지 검증 요청 (Playwright / 미리보기 브라우저).
- **동시에 A가 직접 서버·프로그램을 켜서 작동 확인.**
- 👉 **B 핵심 투입:** A는 "되나/안 되나"를 밖에서 보고, B는 "**왜** 안 되는지"를 코드·로그로 안에서 파악. ⭐

### 7단계. 디버그 (승부처 🔥)
- 첫 결과물은 완벽하지 않다는 전제로 시작.
- 안 되는 부분을 **콕 집어** 스펙 문서 근거로 Claude/Codex에 수정 지시.
- 👉 **B가 길잡이:** "여기 이 변수가 null이라 그래" 같은 포인트를 찍어주면
  AI 수정 지시가 훨씬 정확해진다. → A가 그 포인트를 AI에게 전달.

### 8단계. 발표자료 보강 & 발표 준비
- 완성된 결과물 기준으로 HTML 발표자료 수정·보강.
- 발표 시나리오 정리, 데모 화면 미리 띄워놓기.
- 👉 **주도: A** / B는 기술 질문 대비 (심사위원 Q&A).

## ⏱️ 4시간 압축 가이드

> 평소 워크플로우는 품질이 최고지만 단계가 많다. 4시간엔 아래처럼 줄인다.

| 단계 | 평소 | 4시간 버전 |
|------|------|-----------|
| 스펙 교차검토 | Claude+Codex 풀 검토 | **1회만** + B 빠른 검토 |
| 계획 문서 | 별도 문서 + 교차검토 | 스펙에 짧게 합치거나 **생략** |
| 구현 검증 | Playwright + 직접 실행 | **직접 실행 위주** (Playwright 세팅 시간 절약) |

**아낀 시간은 전부 7단계(디버그)에 몰아준다.** 거기가 승부처.

### 권장 타임라인

| 시간 | 작업 |
|------|------|
| **0:00 – 0:30** | 1~2단계: 기획 + 스펙 확정 (범위 자르기가 제일 중요) |
| **0:30 – 1:00** | 3단계: 간단한 계획 + A는 API 사전준비 |
| **1:00 – 3:00** | 4~5단계: Codex 병렬 구현 / A는 발표자료 / B는 코드 모니터링 |
| **3:00 – 3:30** | 6단계: 검증 + 직접 실행 |
| **3:30 – 3:45** | 7단계: 집중 디버그 |
| **3:45 – 4:00** | 8단계: 발표자료 마무리 + 데모 화면 세팅 |

## ✅ 4시간 황금 규칙

1. **기능은 1개, 많아야 2개.** "이거 하나는 확실히 된다"가 어설픈 3개보다 무조건 이긴다.
2. **자주, 일찍 합친다.** 마지막에 몰아서 합치면 충돌로 망한다.
3. **가짜 데이터 OK.** 진짜 연동 안 되면 하드코딩 데이터로 데모만 돌아가게.
4. **데모가 평가다.** 안 보이는 곳(로그인 등)에 시간 쓰지 말 것.

## 💬 한 줄 요약

> **운전(기획·AI 오케스트레이션)은 A, 정비(검토·디버그)는 B.**
> B의 "코드 읽기"를 디버그(6~7단계)와 스펙 현실성 검토(2~3단계)에 집중시키면,
> 워크플로우의 가장 약한 고리(디버그)가 가장 강한 무기가 된다.
