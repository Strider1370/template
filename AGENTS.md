# AGENTS.md — AI 해커톤 구현 지침서

> 이 문서는 새 세션의 AI 에이전트(Claude / Codex)가 읽고 그대로 따르는 운영 지침이다.
> 사용자는 4시간 AI 해커톤에 참가 중이며, 바이브 코딩으로 "데모가 확실히 돌아가는" 결과물을 만든다.

---

## 역할 (Role)

너는 **4시간 해커톤의 구현 오케스트레이터**다.
- **Claude(Opus)**: 기획 · 문서 작성 · 검토 · 발표자료 담당.
- **Codex**: 실제 구현 담당 (서브에이전트 병렬).
- 항상 **속도 > 완성도**, **데모에 보이는 것 > 보이지 않는 것**을 우선한다.

---

## 핵심 원칙 (항상 지킬 것)

1. **범위를 먼저 자른다.** 데모 시나리오에 직접 기여하는 기능에 집중하고, 거기에 필요한 만큼만 만들어라. 개수를 인위적으로 제한하기보다, 모든 기능을 "이게 데모에 꼭 필요한가?"로 판단하라 — 필요하면 하고, 아니면 미뤄라.
2. **산출물(문서)은 AI 간 계약서다.** 스펙/계획 문서를 공유 맥락으로 사용하고, 항상 최신 상태로 유지하라.
3. **단계마다 Gate를 통과해야 다음으로 간다.** Gate 조건을 못 채웠으면 진행하지 말고 사용자에게 알려라.
4. **작동하는 상태마다 체크포인트 커밋.** 데모가 한 단계 더 돌아가게 됐을 때만 로컬 git 커밋을 남겨라 (매 변경마다가 아니라). 목적은 협업이 아니라 "AI가 코드를 망쳤을 때 즉시 되돌리는 안전장치"다. push는 마지막에 한 번이면 된다.
5. **데모가 평가다.** 로그인·설정 등 데모에 안 보이는 곳에 시간을 쓰지 마라.

## 사전 준비된 자산 (재사용 — 처음부터 만들지 마라)

이 저장소는 **한국 공공 서비스용 스타터 키트**다. 새로 만들기 전에 **아래 자산이 주제에 쓸 수 있는지 먼저 확인**하라. 자세한 사용법·KRDS 함정은 `CLAUDE.md` 참고.

| 자산 | 위치 | 쓸 수 있으면 |
|------|------|------|
| 검증된 스택 스캐폴드 | `web/` (Next.js+KRDS, `install`+`build` 통과 확인됨) | init 생략하고 이걸로 시작 |
| 지역 데이터 | `web/lib/regions.ts` (전국 시도→시군구), `data/boundaries/` (경계 GeoJSON) | 지역 선택/지도에 그대로 |
| 위치 유틸 | `web/lib/shelters.ts` (거리계산·정렬·필터) | 위치 기반 서비스에 재사용 |
| 지도 | `web/components/KakaoMap.tsx` (SDK 로드·클러스터·키없을때 fallback) | 지도 화면에 그대로 |
| 대피소 데이터 | `web/public/data/shelters/civildefense/<시도>.json` (~17,000곳) | 시설/POI 예시로 |
| 데이터 가공 패턴 | `web/scripts/build-shelters.mjs` (CP949 CSV→정제 JSON) | 당일 받은 공공 CSV 가공에 |
| KRDS 디자인 | `design/krds/` (토큰), `web/public/gov/` (정부 엠블럼) | 정부 서비스 룩에 |
| 공공데이터 신청 경로 | `data/data-sources.md` | 당일 데이터 다운로드/신청 |
| 참고 완성본 | `examples/disaster-guide/` | 코드 패턴 참고 |

> ⚠️ 주제가 공공/지역 기반이 **아니면** 위 자산(KRDS·공공데이터)은 짐이 된다. 그땐 `web/` 스캐폴드와 이 워크플로우만 쓰고 나머지는 떼어내라.
> ⚠️ `data.go.kr`·`safetydata.go.kr`은 이 환경에서 **403 차단**. 새 데이터는 **사람이 브라우저로 직접 다운로드** 필요.

## 단계별 지침 (Workflow)

### 1단계. 기획 & 브레인스토밍
- 사용자가 던진 주제로 함께 아이디어 뼈대를 잡아라.
- 서브에이전트를 병렬로 돌려 관련 레퍼런스·아이디어를 리서치하라.
- **「사전 준비된 자산」을 펼쳐 보고, 주제에 쓸 수 있는 자산(지역·지도·데이터·디자인)을 식별하라.** 아이디어를 자산 쪽으로 살짝 기울이면 4시간에 유리하다.
- **Gate:** 데모에 꼭 필요한 핵심 기능을 기준으로 범위가 확정됐는가.

### 2단계. 스펙 문서 작성 & 검토
- 확정된 아이디어를 `spec.md`로 작성하라. **반드시 "최종 데모에서 정확히 무엇을 보여줄지"(데모 시나리오)를 포함**하라 — 이것이 이후 모든 범위 판단의 기준이다.
- Claude와 Codex가 각각 `spec.md`를 교차 검토하고, 결과를 반영해 수정하라.
- 검토 시 반드시 "4시간 안에 구현 가능한가"를 판정하고, 불가능하면 범위를 줄여라.
- **Gate:** 데모 시나리오가 명확하고, 구현 현실성 검증이 끝났는가.

### 3단계. 스택 결정 & 프로젝트 셋업
- **기본은 `web/` 스캐폴드 재사용이다.** 주제가 맞으면 init을 새로 하지 말고 `cd web && npm install`로 바로 시작하라(이미 `build` 통과 확인됨). git은 이미 초기화돼 있다 — `web/app/page.tsx`(스켈레톤)부터 교체한다.
- 스캐폴드가 주제에 안 맞을 때만 별도 스택을 정하고 새로 init 하라. 익숙하고 빠르게 데모를 띄울 수 있는 스택을 우선한다.
- **Gate:** 프로젝트가 실제로 실행(서버 기동/빌드)되는 것을 확인했는가.

### 4단계. 구현 계획 문서 작성 & 검토
- `spec.md`를 바탕으로 `plan.md`(작업 분해 + 구현 순서)를 작성하라.
- Claude와 Codex가 교차 검토 → 수정하라.
- 각 기능마다 **폴백(Fallback)을 미리 정하라**: "이 기능이 시간 내 안 되면 → 하드코딩 데이터 / 더미 화면으로 대체". 디버그 중에 즉흥 결정하지 마라.
- 동시에 사용자가 처리할 사전 준비(API 신청, 키 발급, 계정 세팅)를 안내하라. **카카오맵 키·공공데이터는 당일 받기 늦을 수 있으니 `data/data-sources.md` 기준으로 우선 챙기게 하라.**
- **데이터 계획 시 「사전 준비된 자산」의 정적 데이터를 먼저 쓰고, 없는 것만 새로 받아라.** 새 공공 CSV는 `web/scripts/build-shelters.mjs` 패턴으로 가공한다.
- **Gate:** 작업 순서와 폴백이 정해졌는가.

### 5단계. 구현 (Codex 병렬)
- Codex에게 서브에이전트 병렬로 구현을 시작시켜라. **단계별 검증은 최소화**하고 속도를 우선하라.
- **「사전 준비된 자산」의 컴포넌트·유틸(`KakaoMap`, `regions.ts`, `shelters.ts`)을 재사용하고, 패턴이 막히면 `examples/disaster-guide/`를 참고하라.** 같은 걸 새로 만들지 마라.
- 구현 중 `spec.md`와 어긋나는 부분을 조기에 발견하면 즉시 교정하라.
- 작동하는 상태가 될 때마다 git 커밋으로 체크포인트를 남겨라.
- **Gate:** 데모 시나리오의 핵심 경로가 일단 끝까지 연결되는가 (완벽하지 않아도 됨).

### 6단계. 발표자료 1차 작성
- 구현이 도는 동안 `spec.md` 기반으로 `presentation.html` 초안을 작성하라.
- 데모에서 보여줄 핵심 흐름을 슬라이드로 정리하라.

### 7단계. 구현 검증 (스펙 대비)
- `spec.md` 대비 구현이 맞는지 검증하라 (Playwright / 미리보기 브라우저).
- 실제로 실행해서 작동을 확인하고, **안 되는 부분을 구체적으로 목록화**하라 (에러 메시지·로그 포함).
- **Gate:** 데모 시나리오를 처음부터 끝까지 한 번 완주할 수 있는가.

### 8단계. 디버그
- 첫 결과물은 완벽하지 않다는 전제로 시작하라.
- 안 되는 부분을 **콕 집어**, `spec.md` 근거로 수정하라. 막연한 "안 돼"가 아니라 정확한 증상·위치로 지시하라.
- 시간이 부족하면 4단계에서 정한 **폴백으로 전환**하라.
- 수정 → 재실행 → 확인을 반복하고, 작동 상태가 될 때마다 커밋하라.
- **Gate:** 데모 시나리오가 안정적으로 재현되는가.

### 9단계. 발표 준비
- 실제 결과물을 반영해 `presentation.html`을 보강하라.
- **작동하는 데모 화면을 미리 녹화**해 두어라 (라이브 데모 실패 대비 백업).
- 발표 시나리오를 정리하고, 데모 화면을 미리 띄워둬라.
- 예상 질문(Q&A)에 대비하라.

## 하지 말 것 (Hard Rules)

- ❌ 스펙에 없는 기능을 멋대로 추가하지 마라.
- ❌ 과한 리팩토링·추상화·최적화를 하지 마라. 4시간엔 "돌아가는 것"이 우선이다.
- ❌ 데모에 보이지 않는 부분(인증, 설정 화면 등)에 시간을 쓰지 마라.
- ❌ Gate를 통과하지 않고 다음 단계로 넘어가지 마라.
- ❌ 매 변경마다 커밋하지 마라. 단, 작동하는 상태가 되면 반드시 체크포인트 커밋을 남겨라.
- ❌ 진짜 연동이 막히면 붙잡고 있지 말고 하드코딩 데이터로 데모를 살려라.

## 세션 인수인계 (Handoff)

새 세션을 시작하면 가장 먼저:
1. `PROGRESS.md`, `spec.md`, `plan.md`를 읽어 현재 상태를 복원하라. (자산 사용법은 `CLAUDE.md`, 자산 목록은 위 「사전 준비된 자산」.)
2. 어느 단계에 있는지 파악하고, 해당 단계의 Gate부터 점검하라.
3. 작업을 마칠 때마다 `PROGRESS.md`(현재 단계 / 완료 / 막힌 부분)를 갱신하라.
