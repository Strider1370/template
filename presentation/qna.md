# 예상 Q&A — 임신·출산 지원의 시계

> Stage 08 산출물. 차별점·AI 필요성·구현 한계·데이터 출처·확장/책임 대비. 정직성: manifest 기준.

## A. 차별점 — "정부24 맘편한임신/행복출산이 이미 하지 않나요?"
정부24는 **신청 창구**입니다(무엇을 신청할지는 사용자가 알아야 함). 저희는 *무엇을 언제* 신청할지 판단해 주는 **시간축 레이어**예요. 맘편한임신(임신)·행복출산(출산)이 분절돼 있고, 둘 다 "지금→곧" 시간순 정렬·마감 손실 경고·거주지 자동 매칭의 *결합*은 제공하지 않습니다. 데모의 라이브 재정렬이 그 차이를 30초에 보여줍니다.

## B. AI 필요성 — "정렬은 규칙 엔진 아닌가요? AI가 왜 필요하죠?"
맞습니다. 시간축 정렬·D-day는 **규칙(순수 함수)**으로 합니다 — AI를 거기 쓰지 않는 게 오히려 정직합니다. AI가 본질인 지점은 **규칙으로 못 푸는 비정형 교차 상황**입니다: "쌍둥이+고위험+전입 3개월" 같은 자연어 질문을, 검증된 제도 데이터 안에서만 추려 평이하게 답하는 것. mRelief 같은 해외 자격 스크리너의 LLM 업그레이드 포지션입니다.
> (정직) 현재 데모는 API 키 없이 **검증된 답변(fixture)+규칙 폴백**으로 동작합니다. 키를 연동하면 동일 구조에서 실시간 LLM으로 승격됩니다(서버 프록시에 제도 JSON 주입, 데이터셋 밖 수치 생성 차단 프롬프트).

## C. 데이터 출처·최신성 — "그 60일·금액, 정확한가요?"
모든 핵심 수치는 **정부 1차출처로 검증**했습니다(`research/verified-figures.md`): 부모급여·첫만남 60일 소급(정책브리핑·복지로), 첫만남 200/300만원, 부모급여 0세 100·1세 50만원, 다태아 진료비 태아당 100만원, 산모신생아 40일전~30일후·현행 안내 기준 중위소득 150%, 고위험 90%·300만원, 아동수당 2026년 9세 미만 확대. 데이터에 `verified`·기준일을 달고, 정기 갱신은 운영 과제로 둡니다.

## D. 오안내 책임 — "잘못 안내하면 책임은?"
자격 *판단 보조*까지입니다. 확정·신청은 공식 창구로 연결하고, 데이터셋 밖은 "모른다"고 답하도록 설계했습니다(환각 차단). 최종 책임 주체는 사용자+공식 기관입니다.

## E. (Q&A용) 확장 — "다른 분야로 확장 가능한가요?"
같은 "라이프이벤트 시간축 + 마감 손실 경고" 메커니즘은 청년·고령·창업 등 *신청 시점이 중요한* 모든 공공 지원에 이식 가능합니다. 데이터셋만 갈아끼우면 됩니다.

## F. (Q&A용) 데이터 운영 — "데이터는 어떻게 최신으로 유지하죠?"
현재는 검증된 고정 JSON(혼합 전략). 운영 단계에선 부처/지자체 공표 주기에 맞춘 갱신 파이프라인 + `verified` 플래그·기준일 노출로 신뢰성을 관리합니다. 지자체별 차등(출산장려금)은 지역 데이터 확장 영역입니다.

## G. (Q&A용) 사업화/정책 — "정책 적용 가능성은?"
수혜율(take-up) 제고는 저출생 대응의 직접 지표입니다. 기존 포털에 *시간축 안내 레이어*로 얹거나, 행복출산 원스톱과 연계하는 형태가 현실적 적용 경로입니다.
