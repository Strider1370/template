# 🏆 AI 해커톤 바이브 코딩 워크플로우 (4시간)

> **목표:** Claude + Codex를 활용해 4시간 안에 "데모가 확실히 돌아가는" 결과물을 만든다.

---

## 🛠️ 사용 도구

- **Claude (Opus)** — 기획, 브레인스토밍, 문서 작성, 검토, 발표자료
- **Codex** — 실제 구현 (서브에이전트 병렬)
- **서브에이전트 병렬 실행** — 리서치 / 구현 속도 확보
- **검증** — Playwright 또는 Claude/Codex 앱 미리보기 브라우저 + 직접 실행

---

## 📋 단계별 워크플로우

### 1단계. 기획 & 브레인스토밍
Claude(Opus)에게 주제를 던지고 함께 아이디어 뼈대를 잡는다. 동시에 서브에이전트를 병렬로 돌려 관련 아이디어·레퍼런스를 검색·수집한다.

- [ ] Claude에 주제 던지고 브레인스토밍 시작
- [ ] 서브에이전트 병렬 실행해 레퍼런스/아이디어 리서치
- [ ] 핵심 기능 1~2개로 범위 확정 (제일 중요)

### 2단계. 스펙 문서 작성 & 검토
모인 아이디어를 Claude에게 스펙 문서로 정리시킨다. Claude + Codex가 각각 교차 검토하고, 결과를 Claude에 전달해 스펙을 수정한다.

- [ ] Claude에게 스펙 문서 작성 요청
- [ ] Claude + Codex 교차 검토
- [ ] 검토 결과 Claude에 전달 → 스펙 수정
- [ ] "4시간 안에 가능한가?" 현실성 점검

### 3단계. 구현 계획 문서 작성 & 검토
스펙 문서를 바탕으로 Claude에게 구현 계획 문서를 만들게 한다. 다시 Claude + Codex 교차 검토. **이 작업이 도는 동안 API 사용 신청·키 발급 등 사전 준비를 병행**한다.

- [ ] Claude에게 구현 계획 문서 작성 요청
- [ ] Claude + Codex 교차 검토 → 수정
- [ ] (병행) API 사용 신청 / 키 발급 / 계정·환경 세팅

### 4단계. 구현 (Codex 병렬)
Codex에게 서브에이전트 병렬로 구현 시작을 명령한다. **단계별 검증은 최소화**해서 속도를 우선한다. 구현이 도는 동안 발표자료 작업을 병행한다.

- [ ] Codex에 병렬 구현 명령 (검증 최소화, 속도 우선)
- [ ] 구현 진행 상황 모니터링 (스펙과 어긋나는 부분 조기 발견)

### 5단계. 발표자료 1차 작성
구현이 도는 동안 Claude와 함께 HTML 형식 발표자료 초안을 만든다 (스펙 기반). 실제 구현 완료 후 내용을 보강할 예정.

- [ ] Claude와 HTML 발표자료 초안 작성 (스펙 기반)
- [ ] 데모에서 보여줄 핵심 흐름 정리

### 6단계. 구현 검증 (스펙 대비)
Claude/Codex에게 스펙 문서대로 구현됐는지 검증을 맡긴다 (Playwright / 미리보기 브라우저). 동시에 직접 서버·프로그램을 켜서 작동을 확인한다.

- [ ] Claude/Codex에 스펙 대비 검증 요청 (Playwright / 미리보기)
- [ ] 직접 서버·프로그램 실행해 작동 확인
- [ ] 작동 안 하는 부분 목록화

### 7단계. 디버그 🔥
첫 결과물은 완벽하지 않다는 전제로 시작. 안 되는 부분을 **콕 집어** 스펙 문서 근거로 Claude/Codex에 수정을 지시하며 디버그한다.

- [ ] 안 되는 부분을 구체적으로 지칭
- [ ] 스펙 문서 근거로 수정 지시
- [ ] 수정 → 재실행 → 확인 반복
- [ ] 데모 시나리오 처음부터 끝까지 한 번 완주

### 8단계. 발표자료 보강 & 발표 준비
완성된 결과물 기준으로 HTML 발표자료를 수정·보강하고 발표를 준비한다.

- [ ] 실제 결과물 반영해 발표자료 수정·보강
- [ ] 발표 시나리오 정리
- [ ] 데모 화면 미리 띄워놓기
- [ ] 예상 질문(Q&A) 대비

## ⏱️ 4시간 압축 가이드

> 평소 워크플로우는 품질이 최고지만 단계가 많다. 4시간엔 아래처럼 줄인다.

| 단계 | 평소 | 4시간 버전 |
|------|------|-----------|
| 스펙 교차검토 | Claude+Codex 풀 검토 | **1회만** 빠르게 |
| 계획 문서 | 별도 문서 + 교차검토 | 스펙에 짧게 합치거나 **생략** |
| 구현 검증 | Playwright + 직접 실행 | **직접 실행 위주** (Playwright 세팅 시간 절약) |

**아낀 시간은 전부 7단계(디버그)에 몰아준다.** 거기가 승부처.

### 권장 타임라인

| 시간 | 작업 |
|------|------|
| **0:00 – 0:30** | 1~2단계: 기획 + 스펙 확정 (범위 자르기가 제일 중요) |
| **0:30 – 1:00** | 3단계: 간단한 계획 + API 사전준비 |
| **1:00 – 3:00** | 4~5단계: Codex 병렬 구현 / 발표자료 작성 |
| **3:00 – 3:30** | 6단계: 검증 + 직접 실행 |
| **3:30 – 3:45** | 7단계: 집중 디버그 |
| **3:45 – 4:00** | 8단계: 발표자료 마무리 + 데모 화면 세팅 |

## ✅ 4시간 황금 규칙

1. **기능은 1개, 많아야 2개.** "이거 하나는 확실히 된다"가 어설픈 3개보다 무조건 이긴다.
2. **자주, 일찍 합친다.** 마지막에 몰아서 합치면 충돌로 망한다.
3. **가짜 데이터 OK.** 진짜 연동 안 되면 하드코딩 데이터로 데모만 돌아가게.
4. **데모가 평가다.** 안 보이는 곳(로그인 등)에 시간 쓰지 말 것.
