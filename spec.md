## 1. Answer First
첫 임신을 한 임산부에게, **흩어진 정부 지원을 내 임신 주차·내 동네 기준으로 "지금 신청 → 곧 열림" 시간순으로 줄 세우고, 놓치면 사라질 돈을 D-day로 경고**해, "지금 무엇을 해야 하는지"를 한 화면으로 알려준다.

## 2. Problem
- **사용자**: 박지은(32세, 서울, 첫 임신 8주차, 맞벌이) 같은 초임 임산부/예비부모.
- **사용 상황**: 임신 확인 직후·출산 임박·출산 직후처럼 제도 신청 타이밍이 몰리는 시점. 입덧·야근에 치여 알아볼 여유가 없다.
- **현재 방식**: 정부24·복지로·아이사랑·129·맘카페를 각각 뒤지고, 지원별 신청 시기를 스스로 추적한다(`problem-01`, `jtbd-src-05`).
- **현재 방식의 실패**: 채널이 부처별로 갈라져(행안부/복지부) 전체 그림이 안 보이고, 모두 pull 방식이라 "언제 움직여야 하는지"를 아무도 먼저 알려주지 않는다(`gap-01`).
- **실패 비용**: 신청 창을 놓치면 **돈이 사라진다.** 부모급여·첫만남이용권은 **출생일로부터 60일 이내** 신청해야 출생월부터 소급, 초과 시 신청월부터만 지급(`problem-02`, `verify-src-01`, `verify-src-04` — 정부 1차출처 검증완료). 산모·신생아 건강관리는 출산예정 40일 전~출산 후 30일의 좁은 창(`verify-src-11`).

## 3. 최종 Insight
- **Common Assumption**: 임산부가 지원을 못 받는 건 정보가 부족해서다 → 모아주면 된다.
- **Observed Reality**: 정보는 이미 차고 넘치는데도, 신청 시기·소급 마감을 놓쳐 받을 수 있던 돈을 잃는다.
- **Tension**: 정보량과 수혜는 비례하지 않는다 — "무엇이 있는지"를 알아도 "언제 움직일지"를 모르면 지원은 통째로 사라진다.
- **Reframing**: 이건 정보 부족 문제가 아니라 **타이밍(시점) 문제**다. 임산부에게 필요한 건 카탈로그가 아니라 **"지금 뭘 해야 하는지 알려주는 시계"**다.

## 4. JTBD
- **Functional Job**: 내 상황(주차·지역·소득)에서 지금 신청 가능한 지원을 하나도 안 놓치고 제때 받는다.
- **Emotional Job**: "혹시 놓치고 있나"라는 불안을 덜고, "이것만 하면 된다"는 통제감을 느낀다.
- **Social Job**: 정보에 밝은 똑똑한 부모로 보이고 싶다(놓쳐서 손해 본 부모가 되고 싶지 않다).
- **Current Alternative**: 맘카페 검색·정부24·복지로. 시간축이 없고 pull이라 시기를 놓친다.
- **Success Criteria**: "지금 신청할 것 / 마감 임박 / 곧 열릴 것"이 한 화면에 시간순으로 정리되고, 마감 경고를 보고 바로 행동한다.

## 5. AI Leverage
- **AI 없이도 가능한 것**: 시간축 정렬 자체(임신주차/월령 → 제도 window 매칭)는 순수 규칙 함수로 처리.
- **AI로 더 빠르고 싸지는 것**: 정렬된 결과를 개인 상황 맞춤 한 문단으로 평이하게 브리핑.
- **AI 때문에 처음 가능해지는 것**: 규칙으로 못 푸는 **비정형 교차 상황**(예: "쌍둥이 + 고위험 임신 + 3개월 전 전입")을 자연어로 물으면, 검증된 제도 데이터 안에서만 해당 제도를 추려 답한다. ← 본질 지점(데모 cut 금지, P2).
- **AI가 맡는 역할**: 자연어 질의 해석 + 데이터셋 기반 자격 후보 추림/요약.
- **AI가 하지 않는 역할**: 실제 신청·자격 최종확정·법적 판단·데이터셋에 없는 수치 생성(금지).
- **오류 가능성**: 비정형 질의 오해석, 제도 적용 경계 오판.
- **사람이 최종 판단해야 하는 부분**: 실제 자격·신청은 공식 창구에서 본인이 확인(가드레일 §13).

## 6. Differentiation
- **차별화 수준**: 기능 < 흐름 < 시연 < Insight < Narrative 중 **Insight 차별화 + 시연 차별화** 단계. (시간축으로 문제를 다시 보는 관점 + 라이브 재정렬이라는 데모)
- **우리의 차별점(킥)**: 정부24=신청 *창구* / 우리=언제·어디로 갈지 판단하는 **시간축 내비게이션**. 마감을 손실 프레임(D-day)으로 점등.
- **벤치마킹 출처**: 영국 GOV.UK Sure Start의 클레임 윈도우(`overseas-src-10`) + 호주 Services Australia 사전신청 시간축(`overseas-src-11`) + 에스토니아 proactive 능동성(`overseas-src-01`). 자격 추림은 mRelief 스크리너(`overseas-src-07`)의 LLM 업그레이드.
- **이식 범위**: 기능 통째가 아니라 **메커니즘만** — "기한을 1급 정보로" + "시간축 분리" + "최소질문 자격 추림"을 한국 제도에 맞게 축소.

## 7. Solution
- **입력**: 출산예정일(또는 임신 주차) + 거주지(시/도·시군구, `web/lib/regions.ts` 재사용).
- **처리(AI/시스템)**: ① 입력을 표준 시간축(D-day·아기 월령)으로 정규화 → ② 각 제도 `window`와 매칭해 `now`/`soon`/`past` 분류 + 마감 임박 판정(규칙) → ③ (선택) Claude API에 선별 제도 JSON을 컨텍스트로 주입해 개인화 브리핑/자격 Q&A(폴백: 규칙 템플릿).
- **출력**: 시간축 배지 + "지금 받을 수 있어요" / "곧 받게 돼요" 카드 리스트(우선순위 정렬) + 마감 임박 **손실 경고 배지** + AI 한 줄 브리핑.
- **사용자의 다음 행동**: 카드의 공식 신청처 링크로 이동하거나, "가까운 보건소" 보기, 자유 질문 입력.

## 8. 데모 시나리오
1. 박지은이 랜딩에서 **출산예정일과 거주지(서울 ○○구)**를 입력하고 "내 지원금 보기"를 누른다.
2. 결과 화면에 시간축 배지("임신 28주 · D-84")와 함께 **"지금 받을 수 있어요" 카드(임신·출산 진료비 바우처 등)**, "곧 받게 돼요" 카드가 시간순으로 펼쳐진다.
3. 임신 주차 슬라이더를 **출산 직후로 당기면**(라이브) → 카드가 눈앞에서 **재정렬**되고, 첫만남이용권·부모급여 카드에 **"D-N 지나면 소급분 소멸"** 손실 경고가 빨갛게 **점등**한다. ← Wow
4. (P2) 자유 질문 "쌍둥이인데 고위험이고 3개월 전 이사 왔어요"를 입력 → AI가 데이터셋 근거로 해당 제도(다태아 바우처·고위험 의료비 등)만 추려 답한다.
5. (선택) 카드 → 공식 신청처 링크 / "가까운 보건소" → 지도·목록.

**데모 성공 기준:** 1→3 흐름이 한 번에 돌고, 입력 변경 시 카드 재정렬 + 손실 경고 점등이 화면에서 보이면 성공. (키·API 전부 없어도 1→3은 고정데이터+규칙으로 동작)

## 9. Wow Moment
- **무엇이 보이는가**: 임신 주차/거주지 입력값을 라이브로 바꾸는 순간, 지원 카드가 시간순으로 **재정렬**되고 각 카드의 **D-day가 갱신**되며, 마감 임박 제도에 **손실 경고 배지**가 빨갛게 점등한다.
- **검증 계획 — selector (testid 계약, demo.scenario.yaml과 동일·Stage 05 구현/Stage 07 검증 공유)**:
  `[data-testid="timeline-badge"]`(시간축 배지) · `[data-testid="benefit-timeline"]`(타임라인 컨테이너) · `[data-testid="benefits-now"]`(지금 그룹) · `[data-testid="deadline-warning"]`(손실 경고 배지) · `[data-testid="ai-answer"]`(AI 응답).
- **검증 계획 — assertion**: (a) 입력 변경 전후 `[data-testid="benefits-now"]` 내 카드 구성(개수/순서)이 바뀐다. (b) `[data-testid="deadline-warning"]`이 visible이고 텍스트에 `D-`와 `소멸`(또는 `마감`)이 포함된다. (Stage 07 Playwright 검증)

## 10. Impact
- 근거 없는 수치 없음. 방향성으로: **누락 가능성 감소**(시간순 정렬로 놓치는 제도↓), **의사결정 시간 단축**(여러 포털 탐색 → 한 화면), **마감 손실 회피**(소급 마감 경고). 정량 주장은 검증된 출처(`verified-figures.md`)에 한해 발표에 사용.

## 11. Credibility
- **가장 강한 의심**: "정부24 맘편한임신/행복출산이 이미 통합 안내·신청을 하지 않나?"(`domestic-src-01`)
- **답**: 정부24는 신청 *창구*(무엇을 신청할지는 사용자가 알아야 함). 우리는 *언제·어디로* 갈지를 시간축으로 판단해 주는 레이어로 역할이 다르다. 데모에서 라이브 재정렬+마감 경고로 그 차이를 30초 안에 보인다. 수치는 정부 공식 출처로 검증한 것만(`verified-figures.md`) 단언한다.

## 12. Limitation
- **Soft**: 제도 커버리지(데모는 절벽 핵심 3~4건 → 시간 있으면 확대), 지역 데이터(데모 2~3개 지역).
- **Bounded**: 지자체별 상이 항목은 `regionVaries` 플래그로 대표 지역 값만 노출. AI는 데이터셋에 담긴 제도 범위 안에서만 정확.
- **Hard**: 실제 신청·본인인증·자격 최종확정은 구조적으로 못 한다(공식 창구 연결까지).

## 13. Guardrail
- **잘못된 결과 표시**: 데이터에 `asof`/`verified` 배지, 미검증 항목은 "확인 필요" 라벨. AI는 데이터셋 밖이면 "모른다"로 답(환각 차단 프롬프트).
- **최종 책임**: 자격·신청의 최종 확인은 사용자 본인 + 공식 창구.
- **사용 금지 상황**: 법적·의료적 확정 판단 용도로 쓰지 않는다(안내 보조).
- **사람 개입 지점**: 모든 카드는 공식 신청처 링크로 연결, 실제 행동은 사용자가 수행.

## 14. Closing Message
**"받을 수 있는 지원이 아니라, 지금 받아야 하는 지원을 알려줍니다."** (concept.md 마지막 문장과 동일)

## 15. 범위 밖
실제 신청·본인인증·결제(공식 창구 링크로 대체) · 전국 전 지자체 망라 · 제도 전수(절벽 3~4건만) · 로그인/회원/관리자 · 알림 푸시 인프라. 명시적으로 **안 만든다.**

## 16. 4시간 현실성
가능(조건부). 핵심 경로(입력→시간축 정렬→카드+손실경고+규칙 브리핑)는 고정 JSON+순수함수로 4시간 내 가능. 크리티컬 패스 = **제도 데이터셋(절벽 3~4건) 검증·제작**. 가장 위험한 외부 의존성 = 카카오 REST 키(도메인 등록) → **폴백: 고정 시설 목록 + 지도 fallback**, 그리고 Claude 키 → **폴백: 규칙 기반 브리핑 + Q&A fixture**. 둘 다 없어도 Wow(라이브 재정렬+손실경고)는 성립.
