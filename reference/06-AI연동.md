# AI 연동 (키 입력 + 음성) — 레퍼런스

> AI/LLM 기능(텍스트 채팅·음성 Realtime)을 앱에 붙일 때의 표준 방식 정리.
> 두 축: ① **API 키를 앱 런타임에 직접 입력** ② **음성(GPT Realtime)으로 폼 채우기·대화**.
> 이건 사람이 보고 따라 만드는 참고 문서지, 자동 실행 스크립트가 아니다.

---

## 0. 철칙 — AI는 이해·대화·값추출, 판정·계산은 결정론적 코드

가장 중요한 원칙. **LLM이 최종 판정·계산을 자유롭게 하게 두지 마라**(환각 = 사고).

- LLM이 하는 일: **언어 이해 + 자연어에서 값 추출 + 대화(듣고·말하기)**.
- 코드가 하는 일: **판정·계산·검증** — 규칙/결정트리 같은 **결정론적 코드**.
- 예: 사용자의 말에서 "나이 73, 소득 0" 같은 값을 뽑는 건 AI, 그 값으로 "자격 충족 여부"나 "금액"을 정하는 건 코드.

이게 "결정적 안전망" 원칙이다. AI는 입출력 인터페이스, 결론은 코드.

# 파트 A — LLM API 키: "앱에서 직접 입력"

## A-1. 핵심 결정

모든 AI/LLM(텍스트·음성 Realtime) 기능은 **앱 런타임에 사용자가 직접 넣은 키**로 작동한다.
코드·`.env`·저장소·APK·git에 **키를 절대 박지 않는다.**

### 왜
- 데모는 보통 **운영자(본인)가 직접 시연** → 그 자리에서 키만 넣으면 됨. 서버 토큰 통로(ephemeral)를 안 만들어도 돼서 단순.
- 키를 저장소/배포물에 안 박으니 **유출 위험 없음**(그 자리, 메모리에만 존재).

## A-2. UI — 키 입력 위치
- 헤더(또는 우상단)에 **설정(⚙️) 또는 더보기(⋯) 아이콘** 1개.
- 누르면 **키 입력 모달/드로어** 열림: "OpenAI API 키" 입력칸 + 저장 버튼 + (입력됨/없음) 상태 표시.
- 입력하면 그 세션에서 **AI 기능 활성화**. 비우면 폴백 모드.

## A-3. 저장·보안 규칙
- 키는 **메모리 또는 `sessionStorage`** 에만 둔다. 새로고침/종료 시 사라짐.
  (`localStorage` 영구저장은 지양 — 굳이 쓰면 "이 기기에만 저장됨" 경고를 띄운다.)
- **커밋/배포물(.env·APK·git)에 키 금지.** 런타임 입력만.
- 데모 종료 후 그 키는 **폐기(rotate)** 권장.
- ⚠️ 키 입력칸은 누구나 보므로 **운영자 시연 전제**가 기본.
  불특정 다수가 각자 자기 키를 넣는 BYOK도 같은 UI로 가능(키 책임은 각자).

## A-4. 동작 — 기존 헬퍼에 키만 주입
- **텍스트 LLM**: 기존 LLM 헬퍼(예: `web/lib/llm.ts`의 `openaiChat`) + `/api/llm` 라우트를 그대로 쓰되, **런타임 키를 요청에 실어 보낸다**.
  예: 클라이언트가 헤더 `x-llm-key`로 키 전송 → 라우트가 env 대신 그 키 사용 → 키 없으면 `null` → **폴백**(하드코딩 응답).
- **음성 Realtime**: 클라이언트가 그 키로 직접 연결(데모용 A안, 파트 B 참조).
- **키 없을 때도 앱은 폴백으로 완전히 동작**해야 한다(드롭다운 선택 입력·사전작성 설명 등으로 같은 결과를 보여줌).

# 파트 B — 음성 입력: GPT Realtime

> 기능: 말로 양식 자동 채움 + 자연어 입력 보조 + **단계별 양방향 대화**(질문→답→다음 질문).
> 결정: **모델 `gpt-realtime-2`, reasoning effort `low`** (판정은 코드가 하므로 높은 추론 불필요).
> ⚠️ 모델 라인업은 자주 갱신 → 배포 직전 `https://developers.openai.com/api/docs/models`에서 정확한 ID 재확인.

## B-1. 모델
- **`gpt-realtime-2`** (reasoning `low`) — speech-to-speech, 128k 컨텍스트, 추론강도 조절. 음성↔음성 직결.
- 저지연/저가가 필요한 화면은 `mini` 변형 고려. 통역만 필요하면 `gpt-realtime-translate`.
- 한국어: 양호하나 보장 아님 → **실제 화자·실제 잡음으로 사전 테스트**.

## B-2. 키 입력 = 파트 A 그대로 (표준 A안)
- **설정/더보기(⚙️·⋯) 아이콘 → 키 모달**에서 운영자가 그 자리에서 키 입력 → 메모리/sessionStorage 저장 → 그 키로 Realtime 연결. **코드·.env·APK·git에 키 금지.**
  데모는 운영자가 직접 시연하므로 이게 가장 단순.
- (참고 B안 — 불특정 다수가 각자 폰에서 쓸 때만) **임시키(ephemeral)**: 진짜 키는 서버에만 두고, 서버 토큰 발급 엔드포인트(`/api/realtime-token`)가 1분짜리 임시키를 발급 → 앱이 그걸로 WebRTC 연결. 운영자 시연이면 불필요.

## B-3. 연결 — WebRTC + Agents SDK
- 브라우저/모바일은 **WebRTC**를 쓴다 (WebSocket 아님 — 그건 서버-서버용).
- 공식 SDK **`@openai/agents-realtime`**: `RealtimeAgent`(행동·도구) + `RealtimeSession`(연결·오디오·인터럽션·이력). `tool()`로 도구 정의.
- 참고 앱: `github.com/openai/openai-realtime-agents`

## B-4. 양식 채우기 — 함수호출 + strict 스키마
- 필드별 작은 도구로 쪼갠다: `set_field(name, value)` + `get_missing_fields()` + `submit_application()`.
- 모든 도구 **`strict: true`** + JSON 스키마(enum) → 허용값에 딱 맞춰 채움(환각 방지).
  strict 규칙: 모든 object에 `additionalProperties: false`, 모든 property를 `required`로(선택값은 타입에 `null` 허용).
- UI는 **도구호출 이벤트를 구독**해 입력칸에 실시간 반영.
- 한 번에 한 값씩 보수적으로. 숫자·이름·금액 등 고정밀 값은 **도구 호출 전 read-back(되읽기) 확인**.

## B-5. 단계별(양방향) 대화 베스트프랙티스
> 자격 확인이든 신청서 작성이든, 핵심은 "한 단계씩 묻고·확인하고·다음으로" 진행을 **코드가 운전**하는 것.

- **턴 감지: `semantic_vad` + `eagerness: low`** — 침묵이 아니라 내용으로 발화 끝을 판단 → 사용자가 머뭇거려도 안 끊김. (빠른 응답이 꼭 필요한 단계만 `high`.)
- **흐름은 코드 주도**: VAD는 켜되 `turn_detection.create_response = false`로 두고, 코드가 결정트리를 따라 `response.create`로 다음 질문을 유도한다(대화 단계 진행을 코드가 운전).
- **바지인(끼어들기)**: WebRTC는 `output_audio_buffer.clear`로 아직 안 들려준 오디오를 제거.
- **확인은 out-of-band**(`conversation: "none"`)로 본 대화를 오염시키지 않고 검증/확인.

## B-6. 접근성·정확도 (자연어·고령 사용자 대응)
- **이중 확인 채널**: 들은 값을 **화면 표시 + 음성 read-back** → 시각·청각 동시 확인 후 제출.
- 숫자·이름은 한 자리씩/철자로 재확인. 정정은 항상 되돌릴 수 있게.
- rapid-fire(쏟아붓는) 질문 지양, 자연스러운 확인 대화로.

## B-7. ⚠️ APK(WebView) 마이크 권한 — 놓치면 데모날 실패
안드로이드 WebView 마이크는 **이중 권한**이 필요하다:
- 매니페스트: **`RECORD_AUDIO` + `MODIFY_AUDIO_SETTINGS`** (후자 없으면 권한을 줘도 마이크가 WebView로 안 넘어감)
- **HTTPS 필수**
- 네이티브 `WebChromeClient.onPermissionRequest`를 **override**해서 WebView의 권한 요청을 승인
- Capacitor에서 `getUserMedia({ audio: true })`가 매니페스트만으론 `NotAllowedError`가 나는 사례가 많음 → 위 작업이 필수.

## B-8. 데모 안전
- 발표장 소음·wifi 리스크 → **타이핑 폴백 + 녹화 백업**을 반드시 준비. 핵심 1동작만 음성으로 시연.

## B-9. 아키텍처 한 장
```
앱(React, PWA/WebView) ─getUserMedia─WebRTC─▶ gpt-realtime-2 (reasoning low)
   ▲ 임시키(1분, B안일 때만)                    │ ① 음성이해 ② strict 도구로 값추출
서버(진짜 키 + 결정로직) ◀──────────────────────┘   set_field / get_missing_fields
   ├─ /api/realtime-token (임시키 발급, B안일 때만)
   ├─ 판정/계산(fields)   ← 결정론적 코드 (LLM 아님)
   ├─ 데이터 매칭 (번들 데이터, 키 불필요)
   └─ submit_application
```
> A안(운영자 직접 입력)에서는 임시키 발급 통로가 없고, 앱이 런타임 입력 키로 직접 연결한다.
