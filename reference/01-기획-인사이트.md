# 01. 기획·인사이트 레퍼런스

> 해커톤 기획·판단의 **품질 기준 라이브러리**. 한 번에 다 적용하는 매뉴얼이 아니라, 필요한 항목만 골라 읽고 직접 판단해 적용하는 참고서다.
> 사람이 읽고, 스스로 점검하고, 다음 단계로 넘어갈지 판단한다.

전체 사고 흐름:

```text
주제 → 문제 정의 → 관점 전환 → Insight → JTBD → AI Leverage
→ 차별화 → Solution → Demonstration → Impact → Credibility
→ Limitation → Guardrail → Closing
```

한 번에 전부 풀지 않는다. 지금 필요한 부분만 꺼내 쓴다.

---

## 1. 핵심 철학

### 1.1 기능이 아니라 관점으로 차별화한다
같은 주제를 받은 팀은 대개 비슷한 기능(챗봇·추천·분석·신고 도우미·알림 통합)을 만든다. 기능 하나 더 붙이는 것으로는 기억되지 않는다. **좋은 프로젝트는 문제를 다르게 본다.** 관점이 바뀌면 타깃 사용자·솔루션 구조·시연 장면·발표 메시지까지 따라 바뀐다.

### 1.2 "AI를 붙인 것"과 "AI라서 가능해진 것"은 다르다
핵심 질문: **AI가 있기 때문에 무엇이 처음 가능해졌는가?**

| 수준 | 예 |
|------|-----|
| 낮음 | 챗봇·요약·문장 생성 |
| 중간 | 기존 업무를 더 빠르게·싸게·자동으로 |
| 높음 | 사람이 너무 많이 필요해 불가능했던 일 / 경우의 수가 너무 많던 일 / 개별화 비용이 높던 일 / 언어·문맥·상황을 동시에 해석해야 하던 일 / 다수 가설·시나리오·페르소나 병렬 탐색 |

가능하면 **높은 수준**을 지향한다.

### 1.3 '자유 생성'을 결정적 규칙으로 가둔다 (신뢰 원칙)
AI의 강점(임의 입력 해독, 언어→개념 변환)은 살리되, **분류·매칭·사실 판단은 AI 자유 생성에 맡기지 않는다.** 자유 분류는 흔들리고, 엉뚱한 컨텍스트가 끼면 잘못된 결론을 낸다.

- **결정적 규칙으로 보정**: 고빈도·고위험 결정은 대분류 테이블 + 키워드 매핑 등으로 고정한다. 새 영역은 한 줄 추가로 확장.
- **AI 출력까지 스캔**: 입력이 이미지/사진이면 텍스트가 비어 키워드가 안 걸린다 → AI가 만든 요약·분류 텍스트까지 함께 검사한다.
- **출처는 화이트리스트에서만**: 답·링크는 검증된 자료에서만 인용(환각 차단). 근거 없으면 "확인 필요"로 표시.
- **검증된 자원만 연결**: 외부 링크는 실제 접속 확인 후 넣는다(죽은 링크 주의).

심사위원은 "AI가 틀릴 때 어떻게 막는가"를 본다. **"똑똑함"이 아니라 "정직함(틀려도 엉뚱한 결과가 안 나간다)"** 이 차별점이 된다.

### 1.4 발표는 설명이 아니라 '사용 상상'을 만든다
목표는 심사위원을 `평가자 → 관심자 → 잠재 사용자`로 이동시키는 것. 발표가 끝났을 때 "우리 조직/문제에도 쓸 수 있지 않을까?"를 떠올리게 만든다.

## 2. 심사위원 모델 (모든 판단의 기준선)

**심사위원은 피곤하고, 이해하려 애쓰지 않으며, 계속 의심한다.**

- 첫 5초 — 무엇을 만들었는지 이해되어야 한다.
- 첫 30초 — 왜 필요한지 이해되어야 한다.
- 2분 안 — 실제 작동 증거를 봐야 한다.
- 발표 후 — 한 문장으로 기억되어야 한다.

피해야 할 것: 긴 배경 설명 / 기술 용어 선행 / 기능 나열 / 추상적 가치 / 핵심 주장 지연.

**의심 목록**: 왜 필요한가 · 기존 방식은 왜 부족한가 · 왜 AI인가 · 진짜 동작하는가 · 무엇이 다른가 · 한계는 무엇인가 · 확장 가능한가.
→ 낮은 수준 의심은 **발표 안에서 미리 제거**하고, 높은 수준 의심만 Q&A에 남긴다.

## 3. 문제 정의 / 리서치 사고 기준

리서치는 단순 목록이 아니라 다음을 본다.

**주제 분석**: 주제의 표면적 의미 · 주최 측이 실제 기대하는 가치 · 대부분 팀이 고를 접근 5개 이상 · 이미 흔한 아이디어 · 화려하지만 가치 약한 방향 · 실현/시연 가능성의 균형.

**기존 사례(국내·해외)**: 이미 존재하는 기능 · 기관/민간의 차이 · 사용자가 겪는 실제 마찰 · 규제·데이터 제약 · 해커톤에서 중복될 접근. 해외 사례는 기능을 베끼지 말고 **검증된 메커니즘**을 추출하고, 한국 맥락에 이식 가능한 킥과 출처를 기록한다.

**구현 현실성**: 4시간 안에 시연 가능한가 · 실시간 API가 꼭 필요한가 · 정적 데이터/fixture로 대체 가능한가 · 가장 위험한 외부 의존성 · 데모에 안 보이는 구현 · 핵심 경로를 더 짧게 만들 수 있는가.

### 데이터 모집단 원칙 — 전수 우선
- 검색·필터·매칭의 **모집단은 전수(전체 카탈로그)** 여야 한다. "예시 N개"는 화면 표시 개수일 뿐이다.
- 모집단을 샘플 몇 개로 줄이면 실사용이 아니라 시연용 껍데기가 된다.
- 시간 부족 시 데이터를 줄이지 말고 **보여줄 화면 수(기능 범위)** 를 줄인다.

## 4. JTBD (Jobs To Be Done)

답할 질문: 어떤 상황에서 필요한가 · 실제로 해결하려는 일은 무엇인가 · 지금 어떤 대안을 쓰는가 · 그 대안은 왜 부족한가 · 성공했다고 느끼는 기준은 무엇인가.

정리 형식:

```text
Functional Job      기능적으로 해내려는 일
Emotional Job       느끼고 싶은 감정
Social Job          남에게 보이고 싶은 모습
Current Alternative 현재 쓰는 대안
Failure Cost        대안 실패 시 치르는 비용
Success Criteria    성공 판단 기준
```

## 5. Insight 선택

### 5.1 좋은 Insight의 구조
```text
Common Assumption (흔한 전제)
→ Observed Reality (관찰된 현실)
→ Tension (둘 사이의 긴장)
→ Reframing (문제를 다시 보기)
```

`Reframing`이 핵심이다 — 같은 주제를 남들과 다르게 보게 만드는 한 문장.

### 5.2 후보 생성과 평가
최소 5개 후보를 만들고 각각 평가한다: 새로움 · 설득력 · 근거 가능성 · AI 연결성 · 시연 가능성 · 4시간 구현 가능성 · 위험한 전제.

### 5.3 Insight가 아닌 것
다음은 Solution이나 Benefit일 뿐 관점이 아니다.
- "AI로 더 편리하게" · "정보를 한곳에 모은다" · "맞춤형 서비스" · "사회적 가치를 높인다"

### 5.4 방향 선택 (승인이 필요한 지점)
최종 방향을 2~3개로 압축해 사용자에게 제시한다. 각 선택지의 **의미·예상 결과·되돌릴 수 있는지**를 쉬운 말로 설명하고, 추천이 있으면 이유를 댄다. **사용자가 고르기 전에는 어느 방향도 확정본으로 처리하지 않는다.**

자기검증: Insight가 수사적 문장에 그치진 않는가 · Solution을 Insight처럼 포장하진 않았나 · AI 연결이 억지인가 · 사용자 승인 없이 확정하려 하진 않는가.

## 6. Spec 확정 사고 기준

확정된 내용만 넣는다. **제외**: 리서치 원문 · 탈락한 Insight 후보 · 상세 구현 계획 · 발표 전체 스크립트 · 레이아웃 메타데이터.

### 6.1 Answer First
한 문장: **누구를 위해 / 무엇을 / 어떻게 해결하는가.**

### 6.2 Problem
사용자 · 사용 상황 · 현재 방식 · 현재 방식의 실패 · 실패 비용.

### 6.3 AI Leverage
반드시 셋을 구분한다.
```text
AI 없이도 가능한 것
AI로 더 빠르고 싸지는 것
AI 때문에 처음 가능해지는 것
```
그리고 명시: AI가 맡는 역할 · AI가 하지 않는 역할 · 오류 가능성 · 사람이 최종 판단하는 부분.

### 6.4 Differentiation (킥)
차별화의 강도는 아래로 갈수록 높다. 가능하면 상위를 노린다.
```text
기능 차별화
< 사용자 흐름 차별화
< 시연 차별화
< Insight 차별화
< Narrative 차별화
```

### 6.5 Solution
`입력 → AI/시스템 처리 → 출력 → 사용자의 다음 행동`. 추상 흐름만 적지 말고 **이 화면에 실제 보이는 입력 요소(필드·컨트롤)와 출력 요소(결과 표시)를 구체적으로** 나열한다. 데모 시나리오·Wow Moment와 일치시킨다.

### 6.6 Demonstration
기능 목록이 아니라 **하나의 짧은 이야기**: 구체적 사용자 · 상황 · 입력 · 변화 · Wow Moment · 성공 기준 · 실패 시 폴백.

### 6.7 Impact
근거 없는 숫자를 만들지 않는다. 정량 근거가 없으면: 단계 감소 · 검토 범위 증가 · 개인화 수준 상승 · 의사결정 시간 단축 가능성 · 누락 가능성 감소 등으로 표현.

### 6.8 Credibility
가장 강한 의심 하나를 먼저 다룬다: 데이터 근거 · 실제 작동 증거 · 제한된 사용 범위 · 사람 검토 지점.

### 6.9 Limitation & Guardrail
한계는 세 수준으로: `Soft / Bounded / Hard`.
Guardrail: 잘못된 결과를 어떻게 표시하는가 · 최종 책임은 누구인가 · 언제 쓰면 안 되는가 · 사람이 개입하는 지점은 어디인가.
**핵심 경로는 AI 자유분류에 맡기지 않는다** — 결정적이어야 하는 것은 규칙으로 고정한다(§1.3).

## 7. 발표 내러티브

발표는 **실제 구현 상태**를 기준으로 한다. 구현되지 않은 기능은 확정적으로 주장하지 않는다(mocked/fallback을 완전 구현처럼 말하지 않는다).

권장 흐름:
```text
Answer First → Problem → Insight → Solution → Demo
→ Mechanism → Impact → Limitation/Guardrail → Expansion → Closing
```

### 7.1 시간 배분 (5분 기준)
| 구간 | 시간 |
|------|------|
| Answer First + Problem | 40~50초 |
| Insight + Solution | 40~50초 |
| **Demo + Wow Moment** | **최소 2분** |
| Mechanism | 20~30초 |
| Impact + Guardrail | 30~40초 |
| Closing | 15~20초 |

### 7.2 발표에서 제거할 질문 / Q&A에 남길 질문
- **발표 안에서 미리 답할 것**: 무슨 서비스인가 · 사용자는 누구인가 · 왜 AI인가 · 실제로 작동하나 · 기존과 뭐가 다른가.
- **Q&A에 남겨도 좋은 것**: 도입 시 책임 구조 · 다른 기관 확장 방법 · 데이터 운영 체계 · 장기 사업화/정책 적용.

### 7.3 문장 규칙
짧게 · 기능명보다 사용자 변화 · 첫 문장에서 결과물 설명 · 긴 배경설명 금지 · 데모 전환 문장 명시 · 마지막 문장은 최종 Insight와 연결.

## 8. 단계별 자기검증 체크리스트

스스로 점검하고, 통과하면 사람이 판단해 다음으로 넘어간다.

**리서치**: 출처 없는 사례가 있는가 · 단순 기능 목록에 그쳤는가 · 구현 현실성 검토가 빠졌는가.

**Insight 선택**: 수사적 문장에 불과한가 · Solution을 Insight처럼 포장했는가 · AI 연결이 억지인가 · 사용자 승인 없이 확정하려는가.

**Spec**: 후보와 확정안이 섞였는가 · AI가 하지 않는 역할이 명시됐는가 · Wow Moment가 화면에서 검증 가능한 형태로 정의됐는가 · 범위가 4시간을 초과하는가.

**데모 검증**: Wow Moment가 실제 화면에 존재하는가 · 자동 검증(assertion)이 통과하는가 · 캡처에서 Wow Moment가 명확히 보이는가.

**발표 스크립트**: 발표와 실제 구현이 일치하는가 · 낮은 수준 질문이 남았는가 · 데모 시간이 충분한가 · 근거 없는 숫자가 있는가.

**리허설(승인 지점)**: 제한 시간을 초과하는가 · 라이브 데모 실패 시 대체 가능한가 · 심사위원이 기억할 한 문장이 남는가 · 한계와 Guardrail이 신뢰를 높이는 방식으로 제시됐는가.

---

## 9. 금지 사항

- 사용자 승인 전 최종 Insight를 확정하지 않는다.
- AI를 붙였다는 이유만으로 AI Leverage가 있다고 주장하지 않는다.
- 기능 수로 차별화를 설명하지 않는다.
- 구현되지 않은 기능을 발표에 넣지 않는다.
- 근거 없는 수치를 만들지 않는다.
- 한계를 숨기지 않는다.

## 10. 안전·신뢰

- 실제 개인정보를 데모 입력값으로 쓰지 않는다 — 가상 페르소나·익명 fixture 사용.
- 외부 LLM에 민감정보를 보내지 않는다.
- 의료·법률·재난 등 고위험 영역은 AI 역할과 최종 판단 주체를 명시한다.
- 검증 가능한 주장(현황·통계·정책 수치)은 출처와 연결하고, 근거 없는 최신 사실을 만들지 않는다.

## 11. 시간 부족 시 판단 원칙

```text
단계 생략          → X
기능 범위 축소     → O  (데모 경로 유지 + 검증·발표 시간 확보)
```
