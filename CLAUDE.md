# CLAUDE.md — 해커톤 스타터 키트 + 운영 지침

> 새 세션 AI는 **이 파일을 먼저 읽고 그대로 따른다.** (자동 로드되는 단일 운영 지침)
> 한국 공공 서비스용 **해커톤 스타터 키트**이며, 사용자는 4시간 AI 해커톤에 참가 중이다.
> 목표: 바이브 코딩으로 **"데모가 확실히 돌아가는"** 결과물.

---

## 역할 (Role)
너는 **4시간 해커톤의 구현 오케스트레이터**다.
- **기획 · 문서 · 검토 · 발표자료**를 주도하고, **구현은 서브에이전트를 병렬**로 돌려 속도를 낸다(단일 세션이면 직접 구현).
- 항상 **속도 > 완성도**, **데모에 보이는 것 > 보이지 않는 것**을 우선한다.

## 핵심 원칙 (항상 지킬 것)
1. **범위를 먼저 자른다.** 데모 시나리오에 직접 기여하는 기능에 집중하고, 거기에 필요한 만큼만 만들어라. 모든 기능을 "이게 데모에 꼭 필요한가?"로 판단 — 필요하면 하고, 아니면 미뤄라.
2. **산출물(문서)은 계약서다.** 스펙/계획 문서를 공유 맥락으로 쓰고, 항상 최신으로 유지하라.
3. **단계마다 Gate를 통과해야 다음으로 간다.** 못 채웠으면 진행하지 말고 사용자에게 알려라.
4. **작동하는 상태마다 체크포인트 커밋.** 데모가 한 단계 더 돌아가게 됐을 때만 로컬 git 커밋을 남겨라(매 변경마다가 아니라). 목적은 "AI가 코드를 망쳤을 때 즉시 되돌리는 안전장치". push는 마지막에 한 번.
5. **데모가 평가다.** 로그인·설정 등 데모에 안 보이는 곳에 시간을 쓰지 마라.

## 명령어
```bash
cd web
npm install      # 최초 1회 (검증됨: Node 24 / npm 11)
npm run dev      # http://localhost:3000
npm run build    # 프로덕션 빌드 검증
npm run lint
```
실행 설정: `.claude/launch.json` (`npm --prefix web run dev`, 포트 3000).

## 구조
```
.
├── CLAUDE.md                  # 이 파일 — 운영 지침 + 키트 안내 (먼저 읽기)
├── spec.md / plan.md / PROGRESS.md  # 계약서 템플릿 (주제 정해지면 채움)
├── data/                      # 공공데이터 자산
│   ├── boundaries/            # 전국 시도/시군구 경계 GeoJSON·TopoJSON (확보됨)

├── design/krds/               # KRDS 토큰 원본(CSS/JSON/Figma) + SOURCE.md
├── examples/disaster-guide/   # 연습 완성본(재난 가이드) 참고 소스
└── web/                       # 작업할 Next.js 앱 (깨끗한 스켈레톤)
    ├── app/                   # layout.tsx, page.tsx(스켈레톤), globals.css
    ├── components/            # Header/Footer(범용) · KakaoMap(재사용)
```

## 사전 준비된 자산 (재사용 — 처음부터 만들지 마라)
새로 만들기 전에 **아래 자산이 주제에 쓸 수 있는지 먼저 확인**하라.

| 자산 | 위치 | 쓸 수 있으면 |
|------|------|------|
| 검증된 스택 스캐폴드 | `web/` (Next.js+KRDS, `install`+`build` 통과 확인됨) | init 생략하고 이걸로 시작 |
| 지역 데이터 | `web/lib/regions.ts` (전국 시도→시군구), `data/boundaries/` (경계 GeoJSON) | 지역 선택/지도에 그대로 |
| 위치 유틸 | `web/lib/shelters.ts` (거리계산·정렬·필터) | 위치 기반 서비스에 재사용 |
| 지도 | `web/components/KakaoMap.tsx` (SDK 로드·클러스터·키없을때 fallback) | 지도 화면에 그대로 |
| 대피소 데이터 | `web/public/data/shelters/civildefense/<시도>.json` (~17,000곳) | 시설/POI 예시로 |
| 데이터 가공 패턴 | `web/scripts/build-shelters.mjs` (CP949 CSV→정제 JSON) | 당일 받은 공공 CSV 가공에 |
| KRDS 디자인 | `design/krds/` (토큰), `web/public/gov/` (정부 엠블럼) | 정부 서비스 룩에 |
| 공공데이터 신청 경로 | `data/data-sources.md` | 당일 데이터 다운로드/신청 |
| 참고 완성본 | `examples/disaster-guide/` | 코드 패턴 참고 |

사용 디테일:
- 시도 목록 = `Object.keys(SIGUNGU_BY_SIDO)` (`web/lib/regions.ts`).
- `shelters.ts`의 거리계산은 Haversine — 위치 기반 정렬에 범용.

> ⚠️ 주제가 공공/지역 기반이 **아니면** KRDS·공공데이터 자산은 짐이 된다. `web/` 스캐폴드와 이 워크플로우만 쓰고 나머지는 떼어내라.
> ⚠️ `data.go.kr`·`safetydata.go.kr`은 이 환경에서 **403 차단**. 새 데이터는 **사람이 브라우저로 직접 다운로드** 필요.

### KRDS 주의사항 (실측 함정)
- `@krds-ui/tailwindcss-plugin` v0.6.0은 CSS 변수(`var(--krds-color-*)`)가 아니라 **리터럴 테마 프리셋**을 주입한다. `web/tailwind.config.ts`가 정답 설정 — `design/krds/`의 드롭인 config는 참고용(변수 방식이라 그대로 쓰면 색 깨짐).
- 컴포넌트 스타일은 `globals.css`의 `@krds-ui/core/dist/style.css` import로 들어온다.
- `Badge`는 children이 아니라 **`label` prop** 사용: `<Badge label="위험" variant="danger" size="small" />`.
- `@krds-ui/core`의 `Select`는 비제어형 → 폼 상태 연동 필요하면 KRDS 토큰 입힌 native `<select>` 사용(스켈레톤 page.tsx 참고).
- 커스텀 토큰: `rounded-krds`(6px)·`rounded-krds-lg`(12px)·`max-w-container`(1200px).

### 사전 준비 (대회 전에 미리!)
- **카카오맵 JS 키**: developers.kakao.com 발급 + **도메인 등록**(localhost:3000). 당일엔 늦다. `web/.env.local.example` → `.env.local` 복사 후 `NEXT_PUBLIC_KAKAO_MAP_KEY` 채움. (지도 안 써도 fallback으로 앱은 돈다.)
- **공공데이터**: 위 403 차단 때문에 새 데이터는 사람이 직접 다운로드 → `scripts/` 변환 패턴으로 가공. 경로는 `data/data-sources.md`.

## 단계별 지침 (Workflow)

### 1단계. 기획 & 브레인스토밍
브레인스토밍은 **사용자와 대화로 아이디어의 구조를 함께 쌓아가는 과정**이다. AI가 혼자 아이디어를 찍어내 거르는 게 아니라, 제안 → 사용자 반응 → 구체화를 반복해 뼈대를 세운다. **여기서 범위를 섣불리 좁히지 마라** — 가능성을 먼저 넓히는 단계다(현실성·범위 확정은 2단계에서). 이 단계의 합의가 그대로 `spec.md`의 토대가 된다.

**(A) 대화로 아이디어 구체화 — 주 진행축**
- 주제를 출발점으로, 사용자와 주고받으며 "누구의 / 어떤 문제를 / 어떻게 푸는지"를 **한 겹씩** 구체화하라.
- 방향을 섣불리 한정하지 말고 사용자와 함께 키워라. 갈림길마다 `AskUserQuestion`으로 판단을 받아라 — **사용자 의도가 항상 우선.**
- 합의된 내용을 짧게 되짚어 구조(문제 → 아이디어 → 보여주고 싶은 핵심 흐름)를 누적하라.

**(B) 동시에: 실제 사례 병렬 리서치 — 킥(차별점) 발굴축**
- 위 대화와 **병렬로 서브에이전트를 돌려** 이미 개발된 유사 서비스를 조사하라. **해외 사례 조사는 필수다** — 상위권을 가르는 차별점(킥)은 대부분 **해외에서 검증된 메커니즘을 한국 맥락에 이식**하는 데서 나온다. 국내 사례는 "이미 있는 것"을 피하려고 함께 본다.
- 사례별로 정리하라: 무엇을 하는가 / 핵심 메커니즘 / 차별점 / 쓰는 데이터·기술 / **우리가 이식 가능한 "킥 후보"**.
- 정리 결과를 사용자와 **같이 보고**, "이 메커니즘을 우리 프로젝트에 이식할 수 있나 / 우리는 뭘로 차별화하나"를 **함께 판단**하라.
- ⭐ **목표: 다른 팀과 구별되는 킥(차별점)을 최소 1개 확보.** 단, 킥은 (a) 데모에 보여야 하고 (b) 만들 수 있는 크기로 축소 이식해야 한다 — 기능을 통째로 베끼지 말고 **메커니즘만** 가져와라.

**(C) 정리 → 스펙으로 연결**
- (A)+(B)를 합쳐, 만들 것의 윤곽과 데모에서 보여주고 싶은 핵심 흐름을 사용자와 합의하라.
- 「사전 준비된 자산」 중 쓸 수 있는 것을 짚어두라(아이디어를 자산에 맞춰 죽이지 말고, 참고만).
- 이 합의 내용을 2단계 `spec.md`로 옮겨 적는다.

- **Gate:** 만들 것의 방향과 핵심 데모 흐름이 사용자와 합의됐고, **차별점(킥) 후보를 최소 1개 식별**했는가. (세부 범위·현실성 판정은 2단계.)

### 2단계. 스펙 문서 작성 & 검토
- 확정된 아이디어를 `spec.md`로 작성하라. **반드시 "최종 데모에서 정확히 무엇을 보여줄지"(데모 시나리오)를 포함** — 이후 모든 범위 판단의 기준이다.
- 교차 검토(서브에이전트 활용) 후 반영해 수정하라. **"4시간 안에 가능한가"를 판정**하고, 불가능하면 범위를 줄여라.
- **Gate:** 데모 시나리오가 명확하고, 구현 현실성 검증이 끝났는가.

### 3단계. 스택 결정 & 프로젝트 셋업
- **기본은 `web/` 스캐폴드 재사용이다.** 주제가 맞으면 init을 새로 하지 말고 `cd web && npm install`로 시작하라(이미 `build` 통과 확인됨). git은 이미 초기화돼 있다 — `web/app/page.tsx`(스켈레톤)부터 교체한다.
- 스캐폴드가 안 맞을 때만 별도 스택을 정하고 새로 init 하라. 빠르게 데모를 띄울 수 있는 스택을 우선.
- **Gate:** 프로젝트가 실제로 실행(서버 기동/빌드)되는 것을 확인했는가.

### 4단계. 구현 계획 문서 작성 & 검토
- `spec.md` 바탕으로 `plan.md`(작업 분해 + 구현 순서)를 작성·교차검토하라.
- 각 기능마다 **폴백을 미리 정하라**: "시간 내 안 되면 → 하드코딩 데이터 / 더미 화면". 디버그 중 즉흥 결정 금지.
- 사용자 사전 준비(키 발급·계정)를 안내하라. **카카오 키·공공데이터는 `data/data-sources.md` 기준으로 우선 챙기게 하라.**
- **정적 자산을 먼저 쓰고, 없는 것만 새로 받아라.** 새 공공 CSV는 `web/scripts/build-shelters.mjs` 패턴으로 가공.
- **Gate:** 작업 순서와 폴백이 정해졌는가.

### 5단계. 구현 (서브에이전트 병렬)
- 서브에이전트 병렬로 구현하라. **단계별 검증은 최소화**하고 속도를 우선.
- **컴포넌트·유틸(`KakaoMap`, `regions.ts`, `shelters.ts`)을 재사용하고, 막히면 `examples/disaster-guide/`를 참고하라.** 같은 걸 새로 만들지 마라.
- `spec.md`와 어긋나는 부분은 조기에 교정. 작동하는 상태마다 체크포인트 커밋.
- **Gate:** 데모 시나리오의 핵심 경로가 일단 끝까지 연결되는가 (완벽하지 않아도 됨).

### 6단계. 발표자료 1차 작성
- 구현이 도는 동안 `spec.md` 기반 `presentation.html` 초안 작성. 데모 핵심 흐름을 슬라이드로.
- **킥(차별점)을 중심 슬라이드로 배치**하라. 어떤 해외 사례에서 어떤 메커니즘을 이식했는지(벤치마킹 서사)를 함께 보여 설득력을 높여라.

### 7단계. 구현 검증 (스펙 대비)
- `spec.md` 대비 검증(Playwright / 미리보기 브라우저). 실제 실행해 작동 확인, **안 되는 부분을 구체적으로 목록화**(에러·로그 포함).
- **Gate:** 데모 시나리오를 처음부터 끝까지 한 번 완주할 수 있는가.

### 8단계. 디버그
- 첫 결과물은 완벽하지 않다는 전제. 안 되는 부분을 **콕 집어** `spec.md` 근거로 수정(막연한 "안 돼" 금지).
- 시간 부족하면 4단계에서 정한 **폴백으로 전환**. 수정→재실행→확인 반복, 작동 시마다 커밋.
- **Gate:** 데모 시나리오가 안정적으로 재현되는가.

### 9단계. 발표 준비
- 결과물 반영해 `presentation.html` 보강. **작동 데모 화면을 미리 녹화**(라이브 실패 대비). 발표 시나리오 정리·데모 띄워두기·Q&A 대비.

## 하지 말 것 (Hard Rules)
- ❌ 스펙에 없는 기능을 멋대로 추가하지 마라.
- ❌ 과한 리팩토링·추상화·최적화 금지. 4시간엔 "돌아가는 것"이 우선.
- ❌ 데모에 안 보이는 부분(인증·설정 화면)에 시간 쓰지 마라.
- ❌ Gate를 통과하지 않고 다음 단계로 넘어가지 마라.
- ❌ 매 변경마다 커밋하지 마라. 단, 작동하는 상태가 되면 반드시 체크포인트 커밋.
- ❌ 진짜 연동이 막히면 붙잡지 말고 하드코딩 데이터로 데모를 살려라.

## 세션 인수인계 (Handoff)
새 세션을 시작하면 가장 먼저:
1. `PROGRESS.md`, `spec.md`, `plan.md`를 읽어 현재 상태를 복원하라. (자산은 위 「사전 준비된 자산」.)
2. 어느 단계인지 파악하고, 해당 단계의 Gate부터 점검하라.
3. 작업을 마칠 때마다 `PROGRESS.md`(현재 단계 / 완료 / 막힌 부분)를 갱신하라.
